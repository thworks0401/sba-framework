0. 本書の目的と構成
本書は、SBAフレームワーク設計資料（全14冊）のレビューの結果として特定された「実装フェーズ開始前に確定が必要な事項」「設計上の未定義領域」「技術選定の改善余地」をすべて網羅し、実装担当者が迷わず着手できる状態にすることを目的とする。

1.2 各変更の詳細理由
1.2.1 Vector DB: FAISS → Qdrant（ローカルモード）
FAISSは純粋なANN検索においては最速の部類だが、本システムの中核操作であるSubSkill指定フィルタリング検索に対してネイティブなメタデータフィルタ機能を持たない。SubSkill別の検索はSBAの設計全体で多用されるため、FAISSでこれを実現しようとするとSQLiteへの追加クエリとPython側での後処理が必要になり、実装が複雑化する。
Qdrantのローカルモードはデータをディレクトリとして保存するため、Brain Hot-Swapとの相性が極めて高い。Brain保存時にディレクトリをコピーするだけでVectorインデックスごとバックアップできる。無料・オープンソースであり、Pythonクライアントのみで動作する（サーバープロセス不要）。
【インストール】 pip install qdrant-client  # バージョン 1.9以降推奨
1.2.2 Graph DB: Neo4j → Kuzu（組み込み型グラフDB）
Neo4j Community Editionは機能的には優れているが、本システムへの適用においては以下の問題がある。第一に、Javaランタイムへの依存があり、別プロセスとして起動・管理が必要なため、Hot-Swap時のDB操作（dump/restore）が複雑になる。第二に、Brain Package がディレクトリ単位のシンプルなファイルコピーで管理されるという設計思想と、サーバー型DBの運用モデルが根本的に噛み合わない。
Kuzuは組み込み型（SQLiteと同様の思想）のグラフデータベースであり、データベースの実体がディレクトリとして存在する。これにより Hot-Swap は knowledge_graph/ ディレクトリのコピーで完結し、Javaランタイムも別プロセスも不要になる。クエリ言語はCypherに近いKùzuQLを採用しており、Neo4j向けに書いたクエリのほとんどが最小限の修正で動作する。完全無料・オープンソース。
【インストール】 pip install kuzu  # バージョン 0.6以降推奨
【注意】 Kuzuに変更する場合、設計資料内の「Neo4j dump」という表現は「Kuzuデータベースディレクトリ」と読み替える。brain_packageのファイル名 knowledge_graph.db はそのまま使用可（Kuzuはファイルではなくディレクトリとして保存するため、knowledge_graph/ ディレクトリとする）。
1.2.3 Embeddingモデル: BAAI/bge-m3（CPU動作）
既存設計資料にはEmbeddingモデルの選定が一切記載されていない。FAISSおよびQdrantにベクトルを格納するためには、テキストをベクトルに変換するモデルが必要であり、これが決まらないとLayer1の実装は一行も書けない。
推奨モデルはBAAI/bge-m3（sentence-transformersライブラリ経由、CPU動作）。選定理由は以下の通りだ。第一に多言語対応が強力で、日本語コンテンツ（青色申告・法人営業・各種ドメイン）の埋め込み精度が高い。第二にCPUで動作するため、RTX 3060Ti 8GB VRAMをPhi-4:14Bに専有させる設計と完全に両立できる。第三にモデルサイズが約570MBと小さく、初回ダウンロード後はローカル完結できる。第四にコサイン類似度での重複検出（閾値0.92）にも十分な品質がある。
【インストール】 pip install sentence-transformers  # モデルは初回実行時に自動ダウンロード
モデル	次元数	日本語品質	VRAM	サイズ
BAAI/bge-m3 ✅ 推奨	1024	高（多言語特化）	CPU動作	~570MB
nomic-embed-text（Ollama）	768	中（英語中心）	GPU競合	~274MB
paraphrase-multilingual-MiniLM-L12-v2	384	中	CPU動作	~471MB
1.2.4 音声変換: Whisper → faster-whisper
faster-whisperはOpenAIのWhisperモデルをCTranslate2で再実装したもので、同一モデル・同一精度でオリジナルの約4倍高速に動作する。VRAM使用量はオリジナルと同等（medium モデルで約2.5GB）であり、排他制御の設計変更は不要。pip install faster-whisper で導入可能、完全無料・オープンソース。
1.2.5 CLIフレームワーク: Typer（新規追加）
既存設計資料にCLIの実装フレームワーク指定がなかった。推奨はTyperで、Python型ヒントベースでコマンドと引数を定義でき、argparseに比べ可読性・保守性が高い。Clickをベースとしているため、sba brain swap / load / list / export のような階層コマンドの実装が自然に書ける。自動的にヘルプ文字列が生成され、PowerShellとの互換性も問題ない。
1.2.6 プロセス常駐: NSSM + APScheduler（新規追加）
自律学習ループの「常時バックグラウンド実行」の実現方法が未定義だった。推奨構成はNSSM（Non-Sucking Service Manager）でPythonプロセスをWindowsサービスとして登録し、システム起動時に自動起動させる方式だ。NSSMはフリーウェアであり、クラッシュ時の自動再起動・ログ出力の管理もサポートする。APSchedulerはそのPythonプロセス内で常時稼働させる。
 
2. Priority 1 — 実装開始前に確定が必要な事項
以下4点は実装フェーズ1（Agent Core + Brain管理基盤）を開始するDay1前に確定しなければならない。いずれかが未確定のままだと、コードの基盤部分から設計が宙に浮く。

2.1 Embeddingモデルと使用方針の確定
モデルはBAAI/bge-m3（CPU動作）に確定する。以下に実装上の標準的な使い方を示す。
# src/sba/utils/embedder.py
from sentence_transformers import SentenceTransformer
import numpy as np

class Embedder:
    MODEL_NAME = "BAAI/bge-m3"
    _instance = None  # シングルトン（モデルロードは重い）

    @classmethod
    def get(cls) -> 'Embedder':
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance

    def __init__(self):
        # device='cpu' でVRAM競合を完全回避
        self.model = SentenceTransformer(self.MODEL_NAME, device='cpu')

    def encode(self, texts: list[str]) -> np.ndarray:
        return self.model.encode(
            texts,
            batch_size=32,
            normalize_embeddings=True,  # コサイン類似度用
            show_progress_bar=False
        )

    def encode_single(self, text: str) -> np.ndarray:
        return self.encode([text])[0]

チャンクテキストをKnowledge Baseへ格納する前に、このEmbedderでベクトル化してQdrantに保存する。类似度判定（重複チェック・コサイン類似度 > 0.92）もこのモデルのベクトルを使用する。

    requires_human_review BOOLEAN NOT NULL DEFAULT false,

    PRIMARY KEY (id)

-- SubSkillNode: SubSkillのハブノード

3.1 プロセス常駐方式（Windows）
自律学習ループはAPSchedulerを使ったPythonプロセスとして常時稼働させる。Windowsでこれを実現するために、NSSM（Non-Sucking Service Manager）を使ってWindowsサービスとして登録する。NSSMはフリーウェアでhttps://nssm.cc/downloadから入手できる。
セットアップ手順（PowerShell）
# 1. NSSMをダウンロードして C:/tools/nssm/ に展開

# 2. SBAをWindowsサービスとして登録
nssm install SBAAgent "C:/Python311/python.exe"

3.2 テキストチャンク分割方針の確定
取得したWebページ・PDF・動画字幕テキストをFAISS（Qdrant）に格納する際のチャンク分割方針を以下に確定する。
パラメータ	確定値	理由
目標チャンクサイズ	400〜600 tokens（≒300〜450字）	検索精度とコンテキスト量のバランス
チャンクオーバーラップ	50 tokens	境界付近の知識断片化を防ぐ
最小チャンクサイズ	50 tokens以下は破棄	短すぎるチャンクは検索ノイズになる
境界優先度	文境界 > 段落 > 文字数上限	自然言語の意味単位を尊重
コードブロック	関数・クラス単位で分割しない	コードの意味的なまとまりを維持
構造化ドキュメント	見出し（h1〜h4）で必ず分割	セクション単位で知識を整理
動画字幕	30〜60秒のセグメント単位で初期分割	話題の切り替わりに沿う

# src/sba/utils/notifier.py
from plyer import notification  # pip install plyer
from loguru import logger
from pathlib import Path

REVIEW_LOG = Path('C:/SBA/logs/human_review.log')

def notify_human_review(subskill: str, chunk_ids: list[str], reason: str):
    """矛盾解消不能・要人間確認を通知する"""
    msg = f'[{subskill}] {reason} | chunks: {chunk_ids}'
    
    # ログファイルへ追記
    with open(REVIEW_LOG, 'a', encoding='utf-8') as f:
        f.write(f'{msg}\n')
    
    # Windowsデスクトップ通知
    notification.notify(
        title='SBA: 人間確認が必要です',
        message=f'[{subskill}] {reason[:80]}',
        app_name='SBAAgent',
        timeout=10
    )
    logger.warning(f'HUMAN_REVIEW_REQUIRED: {msg}')

  keep_versions: 7                   # 直近7世代を保持
  include_active: true               # [active]もバックアップ対象
