## 1. Brain Hot-Swapの概要

### 1.1 Hot-Swapとは
Brain Hot-Swapとは、Agent Coreを停止・再構築することなく、装着中のBrainを保存し、新たなBrainを差し替え、必要に応じて過去のBrainを再装着できる仕組みを指す。[file:1]  
SBAでは、Agent Coreは常に1体・固定で稼働し、差し替える対象はBrainパッケージのみとする。[file:1]

### 1.2 目的
- 複数ドメインのBrainを並行育成できるようにする
- Brain育成状態を安全に保管し、いつでも育成再開できるようにする
- エクスポート前にBrainを安定状態として固定できるようにする

### 1.3 対象範囲
本書で扱うのは、以下に関わる設計である。
- Blank Brain Template
- Brain Bank
- [active] スロット
- swap / load / export 時のI/Oフロー
- バージョン・レベル情報の扱い

## 2. 構成要素

### 2.1 Agent Core（固定側）
- 役割: 推論エンジン（Phi-4:14B）とOrchestratorを含む、SBAの中枢。[file:1]
- 特徴: 常に1体・固定・変更不可。[file:1]
- Hot-Swapの対象外（Agent Coreは入れ替えない）。

### 2.2 Brain（差し替え側）
Brainは以下のファイル群を含むパッケージとして扱う。[file:1]

- knowledge_graph.db（Neo4j dump）
- vector_index.faiss（FAISSインデックス）
- experiment_log.db（SQLite）
- learning_timeline.db（SQLite）
- self_eval.json
- subskill_manifest.json
- metadata.json（domain, version, level など）[file:1]

### 2.3 Blank Brain Template
- 役割: 新規ドメインBrain生成のマスターコピー。[file:1]
- 特徴:
  - 学習データ0・知識0の初期状態。[file:1]
  - self_eval.json は全SubSkillスコア0.0。[file:1]
  - subskill_manifest.json は空配列。[file:1]
  - metadata.json は `{ domain: null, version: "0.0", level: 0 }`。[file:1]
  - 直接更新・上書き禁止（必ず複製して使用）。[file:1]

### 2.4 Brain Bank
- 役割: 育成済みBrainを保管するディレクトリ群。[file:1]
- 内容:
  - 各Brainは `<ドメイン名>_v<バージョン>/` ディレクトリとして格納。[file:1]
  - ディレクトリ単位でコピー・バックアップが可能な形に統一。

### 2.5 [active] スロット
- 役割: 現在装着・育成中のBrainを配置するスロット。[file:1]
- 実体: brain_bank直下の `[active]/` ディレクトリ（シンボリックリンク or コピー）。[file:1]
- Agent Coreは常に [active] 以下のBrainを対象として動作する。

## 3. ディレクトリ構造

### 3.1 ルート構造
標準ディレクトリ構造は以下とする。[file:1]

```text
C:/SBA/brain_bank/
├── _blank_template/
│   ├── knowledge_graph.db
│   ├── vector_index.faiss
│   ├── experiment_log.db
│   ├── learning_timeline.db
│   ├── self_eval.json
│   ├── subskill_manifest.json
│   └── metadata.json
├── Python開発_v1.4/
│   ├── knowledge_graph.db
│   ├── vector_index.faiss
│   ├── experiment_log.db
│   ├── learning_timeline.db
│   ├── self_eval.json
│   ├── subskill_manifest.json
│   └── metadata.json
├── 法人営業_v0.8/
│   └── （同構造）
└── [active]/
    └── （同構造）
```

### 3.2 各ディレクトリの意味
- `_blank_template/` : マスターテンプレート。更新禁止。[file:1]
- `<ドメイン名>_vX.Y/` : 育成済みBrainのスナップショット。[file:1]
- `[active]/` : 現在育成中のBrainスロット。[file:1]

### 3.3 パスの前提
- ルートパス `C:/SBA/brain_bank/` は設定ファイルで変更可能とする（本書ではデフォルト値として記載）。
- OS依存はBrain Package・保存形式設定書で詳細定義する。

## 4. Brain Package仕様（概要）

### 4.1 必須ファイル一覧
各Brainディレクトリに必須となるファイルは以下とする。[file:1]

1. `knowledge_graph.db`
2. `vector_index.faiss`
3. `experiment_log.db`
4. `learning_timeline.db`
5. `self_eval.json`
6. `subskill_manifest.json`
7. `metadata.json`

### 4.2 metadata.json（概要）
最低限保持すべきメタ情報は以下とする。[file:1]

- domain: ドメイン名（例: "Python開発"）
- version: バージョン文字列（例: "1.4"）
- level: 育成レベル（Lv.1〜3）
- last_saved_at: 最終保存日時
- その他、本番仕様は Brain Package・保存形式設定書で詳細定義。

## 5. 状態モデル

### 5.1 Brainライフサイクル
Brainの主な状態は以下とする。

- `Blank`: Blank Templateから複製された直後の状態
- `Active`: [active] スロットに装着され、育成中の状態
- `Saved`: Brain Bankに保管されている状態
- `Exported`: エクスポートディレクトリへ出力された状態

### 5.2 状態遷移の概要
代表的な遷移：

- Blank → Active（新規育成開始）
- Active → Saved（swap時の保存）
- Saved → Active（loadによる再装着）
- Saved → Exported（exportによる出力）

詳細な状態遷移図は別途図解（PlantUML等）で添付してもよい。

## 6. Hot-Swap操作（概念）

### 6.1 swap（保存 + 新規開始）
現在育成中のBrainを保存し、新しいドメイン用のBlank Brainを装着して育成を開始する操作。[file:1]

### 6.2 load（再装着）
Brain Bankに保管されているBrainを[active]へロードし、その続きから育成を再開する操作。[file:1]

### 6.3 list（一覧）
Brain Bankに存在するすべてのBrainと、[active] の状態を一覧表示する操作。[file:1]

### 6.4 export（エクスポート）
指定したBrainをSBA外部で利用可能な形式として出力する操作。[file:1]

## 7. CLIインターフェース設計（概要）

※詳細な引数仕様やオプションはCLI仕様書で定義し、本書では役割と内部フローのみを扱う。

### 7.1 sba brain swap

#### 7.1.1 目的
- 現在のActive BrainをBrain Bankへ保存する
- 新しいドメインのBlank Brainを生成しActiveに装着する

#### 7.1.2 使用例
```bash
sba brain swap --save "Python開発_v1.4" --new "法人営業" --from-blank
```

#### 7.1.3 内部処理フロー（概要）
1. `[active]/` の全ファイルを `brain_bank/Python開発_v1.4/` にコピー保存。[file:1]  
2. 保存先 `metadata.json` の `version`, `last_saved_at` を更新。[file:1]  
3. `[active]/` ディレクトリを空にする。[file:1]  
4. `_blank_template/` を `brain_bank/法人営業_v0.0/` としてコピー生成。[file:1]  
5. `法人営業_v0.0/` を `[active]/` にコピー。[file:1]  
6. `subskill_manifest.json` に法人営業ドメインのSubSkill構造を自動生成・書き込み。[file:1]  
7. Ollamaモデルを再起動し、Brainコンテキストをリセット。[file:1]

---

### 7.2 sba brain load

#### 7.2.1 目的
保管済みのBrainをActiveスロットに再装着し、育成を再開する。[file:1]

#### 7.2.2 使用例
```bash
sba brain load "Python開発_v1.4"
```

#### 7.2.3 内部処理フロー（概要）
1. 現在の `[active]/` 状態を自動保存（上書き確認プロンプト付き）。[file:1]  
2. `brain_bank/Python開発_v1.4/` を `[active]/` にコピー。[file:1]  
3. FAISSインデックスをメモリにロード。[file:1]  
4. Neo4jへDBをリストア。[file:1]  
5. `self_eval.json` を読み込み、SubSkill評価状態を復元。[file:1]  
6. 育成ループを再開。[file:1]

---

Brain Bankに保存されているBrainと、そのバージョン・レベル・サイズ・最終保存日時、および現在のActive Brainを一覧する。[file:1]

## 8. I/O・DB操作の基本方針

### 8.1 セーブ処理の原則
- セーブはディレクトリ丸ごとコピーを基本とし、個別ファイルの書き忘れを防ぐ。
- Neo4j dump、FAISSインデックスファイル、SQLiteファイルはAtomicなコピーを意識する。
- セーブ前後で `metadata.json` のバージョン・日時を必ず更新する。

### 8.2 ロード処理の原則
- ロード前にActive状態の自動保存（ロールバック可能な形）を必須とする。
- ロード後は、FAISS / Neo4j / SQLite / JSON の整合性チェックを行い、破損時は起動を中断する。

### 8.3 Blank Template利用時の注意
- `_blank_template/` は読み取り専用として扱い、直接書き込まない。
- 新規Brainは必ずテンプレートのコピーから作成し、コピー後にSubSkill構造・メタ情報を初期化する。

## 9. バージョンとレベル

### 9.1 バージョン
- `version` はBrainの育成スナップショットごとに更新する。
- 命名例: `v0.0`（新規）、`v0.8`、`v1.4`、`v2.1` など。[file:1]

### 9.2 レベル
- `level` はLv.1〜3の育成レベルを保持する。[file:1]
- 自己評価結果に基づき自動更新されるが、Hot-Swap時点の値をそのまま保存する。

---

## 10. エラー・ロールバック方針

### 10.1 swap失敗時
- 保存先へのコピー中断 / 失敗時は、Activeの状態を保持し、swapをロールバックする。
- 新規Brain生成・コピーが失敗した場合も、Activeは変更しない。

### 10.2 load失敗時
- ロード対象のファイル破損時は、Activeの保存だけ行い、ロード自体はキャンセルする。
- Activeを空のままにしない。

### 10.3 export失敗時
- 部分的なファイル出力にとどまった場合は、出力ディレクトリを削除 or 無効化フラグを付与する。
- Brain Bank側の元データには一切変更を加えない。

## 11. 運用ルール

### 11.1 命名規約
- ディレクトリ名: `<ドメイン名>_v<version>`（例: `Python開発_v1.4`）。[file:1]
- ドメイン名は日本語可とするが、ファイルシステム依存文字は避ける。

### 11.2 定期バックアップ
- Brain Bank全体を定期的にバックアップする。
- 特に`_blank_template/`と主要Brainについては別ストレージ保管を推奨。

### 11.3 同時編集の禁止
- 同一Brainを複数プロセスが同時にActiveとして扱う構成は禁止。
- 常に「1 Agent Core × 1 Active Brain」の組み合わせを維持する。
