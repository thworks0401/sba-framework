# 学習ソース・収集方針設定書（Web / PDF / 動画 / API 収集戦略）

---

## 0. 文書情報

### 0.1 文書名
学習ソース・収集方針設定書（Web / PDF / 動画 / API 収集戦略）

### 0.2 目的
本書は、SBAにおける自律学習のためのデータ取得ソース（Web / PDF / 動画 / API / 構造化データなど）の一覧と、それらをどのような方針・優先度・信頼度で利用するかを定義することを目的とする。[file:1]

### 0.3 本書の位置付け
- 上位資料: 自律学習ループ設定書、SBAコンセプト・境界設定書
- 関連資料: コスト・レート制限運用書、推論エンジン・VRAM運用設定書
- 下位資料: 各ソース別取得モジュール実装仕様書（Pythonコード）

### 0.4 想定読者
- 自律学習ループ実装担当
- Webクローラ / データ取得担当
- 運用担当（APIキー・レート制限管理）

## 1. 基本方針

### 1.1 無料優先・ローカル完結
- 可能な限りローカル・無料ソース（Ollama, FAISS, Neo4j, Whisper, Browser-Use, yt-dlp 等）を優先する。[file:1]
- 外部APIの無料枠は「補助的に」使い、超過による課金発生を避ける。[file:1]

### 1.2 多様な形式の活用
- テキスト（Web/PDF/書籍）、動画・音声、コード、統計データなど、多様な形式のリソースを扱う。[file:1]
- 形式に応じて専用パイプライン（HTML → テキスト、PDF → テキスト、音声 → テキスト 等）を構築する。[file:1]

### 1.3 SubSkill別のソース戦略
- Brain / SubSkillの種類に応じて、ソースの優先度とフィルタリング方針を変える（例: Tech系ならGitHub/Stack Overflowを強め、法律系なら公式法令・ガイドラインを強める）。[file:1]

### 1.4 信頼スコアの一貫性
- すべての取得データに対して、出典に応じた初期信頼スコアを付与し、その後の実験・矛盾検出で更新していく。[file:1]

## 2. 信頼スコアと品質レイヤ

### 2.1 初期信頼スコアの基準
取得元ごとの初期信頼スコア（目安）は以下とする。[file:1]

- 学術論文 / 公的統計: 0.95（arXiv, OpenAlex, e-Stat, World Bank 等）[file:1]
- 公式ドキュメント: 0.90（docs.python.org, MDN, ベンダー公式ドキュメント 等）[file:1]
- 専門サイト / 技術ブログ: 0.80[file:1]
- 一般ブログ / まとめサイト: 0.65[file:1]
- SNS・掲示板・Q&A（Stack Overflow等）: 0.50（要コンテキスト確認）[file:1]

### 2.2 要確認フラグ
- 初期信頼スコアが0.60未満のデータには「要確認」フラグを付与し、重要な結論に使う際は追加検証を必須とする。[file:1]

### 2.3 その後の更新
- 自己実験の成功/失敗、矛盾検出の結果に応じて、Knowledge Base 内の信頼スコアを更新する（詳細は自己実験エンジン・知識統合設計側で定義）。[file:1]

## 3. Web検索・ページ取得戦略

### 3.1 検索エンジン

#### 3.1.1 DuckDuckGo Search（ddg-search）
- 特徴: APIキー不要・無料・検索結果URL一覧取得。[file:1]
- 用途: 一般Web検索のエントリポイント。
- 方針:
  - レート制限なしだが、1秒間隔などのスロットリングを推奨。[file:1]
  - SerpApi等の有料検索APIは、原則使用しない（代替可能）。[file:1]

### 3.2 ページ取得・レンダリング

#### 3.2.1 Jina Reader（r.jina.ai）
- 特徴: URLの前に `https://r.jina.ai/` を付けるだけでMarkdown化、APIキー不要・無料。[file:1]
- 用途: 静的/多くの動的サイトのコンテンツ抽出。
- 方針:
  - まずはJina Readerを試し、失敗した場合のみBrowser-Use/Playwrightへフォールバックする。[file:1]

#### 3.2.2 Browser-Use + Playwright
- 特徴: Python製・自律ブラウザ操作・Playwrightベース、ログインやフォーム入力・SPAにも対応。[file:1]
- 用途: JavaScript必須サイトやログインが必要な管理画面など。
- 方針:
  - 高コストなため、必要な場合に限定して利用する。[file:1]
  - 取得頻度の高い重要ページはキャッシュ・再利用を検討。

#### 3.2.3 BeautifulSoup4
- 用途: 静的HTMLページからのテキスト抽出。[file:1]
- 方針:
  - Jina Readerで十分な場合はそちらを優先し、細かい部品取りが必要な場合に利用する。

## 4. 文書・PDFソース戦略

### 4.1 学術・技術文書

#### 4.1.1 arXiv API
- 用途: 最新学術論文の全文・アブストラクト取得。[file:1]
- 制限: 明確な制限はないが、リクエスト間隔3秒推奨。[file:1]
- 方針:
  - AI/ML/CS系Brainでの深い理論学習に使用。
  - 長文の要約・構造化はGemini 2.5 Flashに任せることも検討。[file:1]

#### 4.1.2 OpenAlex API
- 用途: 学術論文・引用関係・著者情報、登録不要・完全無料。[file:1]
- 方針:
  - arXivと組み合わせて、引用ネットワークをKnowledge Graphに反映する用途で使用する。[file:1]

### 4.2 一般知識・百科事典

#### 4.2.1 Wikipedia API
- 用途: 概念定義・関連情報の取得。[file:1]
- 方針:
  - 基礎的な定義・概念整理の出発点として利用。
  - 詳細・最新動向は別ソース（論文、公式ドキュメント）で補完する。

### 4.3 電子書籍・古典

#### 4.3.1 Project Gutenberg
- 用途: 著作権切れ文書・古典文学・歴史資料のテキスト取得。[file:1]
- 方針:
  - 歴史・文学系Brainや、長文スタイル学習に利用。

### 4.4 PDF解析

#### 4.4.1 PDFMiner / pypdf
- 用途: ローカルPDF・Web上PDFのテキスト抽出。[file:1]
- 方針:
  - 取得後は長文をチャンクに分割し、GeminiまたはTier1で要約・構造化する。
  - PDF構造（見出し・段落）を可能な範囲で維持する。

## 5. 動画・音声ソース戦略

### 5.1 動画ダウンロード・字幕

#### 5.1.1 yt-dlp
- 用途: YouTube・ニコニコ動画等から動画・音声・字幕ダウンロード。[file:1]
- 方針:
  - 字幕が存在する場合は字幕テキストを直接取得（音声変換より優先）。[file:1]
  - 動画本体の保存は最小限にし、テキスト化後はテキスト中心に扱う。

### 5.2 メタデータ・API

#### 5.2.1 YouTube Data API v3
- 用途: 動画メタデータ・字幕一覧・チャンネル情報取得。[file:1]
- 無料枠: 10,000 units/day。[file:1]
- 方針:
  - 動画検索・フィルタリングに利用し、字幕取得自体はyt-dlpを優先する。[file:1]

### 5.3 音声→テキスト

#### 5.3.1 Whisper（OpenAIモデル・ローカル）
- 用途: 字幕なし動画・音声ファイルのテキスト変換。[file:1]
- 実行環境: RTX3060Ti + CUDA、mediumモデル推奨（精度とVRAMのバランス）。[file:1]
- 注意:
  - Tier1（Phi-4:14B）とのVRAM競合のため、同時起動禁止。[file:1]
  - Whisper使用時はOllamaモデルをアンロードしてから実行する排他制御が必要。[file:1]

## 6. コード・技術情報ソース戦略

### 6.1 GitHub API
- 用途: README・コード・Issue・Wikiの取得。[file:1]
- 無料枠: 認証あり5,000 req/hour、匿名60 req/hour。[file:1]
- 方針:
  - Tech系Brain（Python開発, DevOps等）の実装パターン学習に利用。
  - Rate-limitヘッダを見てスロットリングする。

### 6.2 Stack Overflow API
- 用途: 実務的な問題解決パターン・ベストプラクティス取得。[file:1]
- 無料枠: 10,000 req/day。[file:1]
- 方針:
  - Q&Aのペアとして取得し、「問題→解決策」の構造をSubSkillに反映。
  - 個々の回答の投票数・受諾フラグも信頼度の補助指標とする。

### 6.3 PyPI JSON API
- 用途: Pythonライブラリの仕様・バージョン・説明取得。[file:1]
- 方針:
  - ライブラリ選定、依存関係、バージョン互換性に関する知識として格納。

### 6.4 MDN Web Docs API
- 用途: Web技術仕様の公式ドキュメント取得。[file:1]
- 方針:
  - フロントエンド・Web系Brainにおける基礎仕様の参照先として利用。

## 7. 専門API・ニュース・統計データ戦略

### 7.1 LLM補完API

#### 7.1.1 Gemini 2.5 Flash API
- 無料枠: 15 req/min・1,500 req/day・1M tokens/min。[file:1]
- 主用途: 大量テキストの要約・構造化・Tier1の補完処理。[file:1]
- 方針:
  - token数8,000超やTier1待機10秒超過時のオフロード先として使用（エンジン切替ロジックに準拠）。[file:1]
  - デイリーカウンタで残枠を管理し、残100を切ったら当日中の新規呼び出しを停止する。[file:1]

### 7.2 モデル・ペーパー情報

#### 7.2.1 Hugging Face Inference API
- 無料枠: 月30,000リクエスト。[file:1]
- 用途: モデル・データセット・論文情報取得。
- 方針:
  - モデル・データセットカタログ的な知識としてBrainに格納。

### 7.3 ニュース・時事情報

#### 7.3.1 NewsAPI
- 無料枠: 100 req/day。[file:1]
- 方針:
  - 時事情報が重要なBrainのみで限定的に利用。
  - 優先度は低め（他ソースで代替できる場合はそちらを優先）。[file:1]

### 7.4 気象・環境・統計系

#### 7.4.1 Open-Meteo API
- 用途: 気象・環境データ取得、完全無料。[file:1]

#### 7.4.2 e-Stat（日本政府統計） / World Bank API
- 用途: 日本・世界の統計データ取得。[file:1]
- 方針:
  - 経済・社会系Brainの定量的根拠データとして利用。

## 9. Brain / SubSkill別戦略

### 9.1 Tech / 開発系 Brain
- 優先ソース:
  - GitHub API / Stack Overflow / PyPI / 公式ドキュメント / MDN / arXiv（CS）[file:1]
- 二次ソース:
  - 技術ブログ、カンファレンス動画（yt-dlp+Whisper）

### 9.2 ビジネス / 経営 / マーケ系 Brain
- 優先ソース:
  - 公式ガイドライン、統計データ（e-Stat / World Bank）、専門書籍PDF、信頼性の高いレポート。
- 二次ソース:
  - ビジネスブログ、事例記事、ニュース。

### 9.3 法律 / コンプライアンス系 Brain
- 優先ソース:
  - 法令集、行政機関公式サイト、ガイドラインPDF。
- 二次ソース:
  - 解説記事（要低信頼フラグ）、論文。

※実際のBrainカテゴリごとの具体戦略は、Brainカタログと連携して別表に落とし込む。

## 10. レート制限・コスト制御との連携

### 10.1 日次カウンタ管理
- すべての外部API呼び出しは、SQLiteベースの日次カウンタで管理し、上限前に自動スロットリングする。[file:1]

### 10.2 有料発生回避ルール（要約）
- Gemini API 残枠が100を切ったら当日中の新規呼び出しを停止。[file:1]
- DuckDuckGoで代替できる場合はSerpApi等の有料検索は使わない。[file:1]
- YouTube字幕はyt-dlpで無料取得、APIは補助のみ。[file:1]

## 11. 実装・I/O共通方針

### 11.1 クリーニング標準
- 各ソースから取得したデータは、共通クリーニングパイプライン（広告除去・正規化・チャンク分割）を通してからKnowledge Baseへ渡す。[file:1]

### 11.2 メタデータ付与
- すべての取得データに、少なくとも以下を付与する:
  - ソース種別
  - URL / ファイルパス
  - 初期信頼スコア
  - 取得日時
  - 取得元API名（任意）[file:1]

### 11.3 Learning Timelineとの連携
- 学習履歴（Layer4）には、ハッシュ値・ソース種別・SubSkillタグ・鮮度スコアを記録し、重複防止と鮮度管理に活用する。[file:1]
