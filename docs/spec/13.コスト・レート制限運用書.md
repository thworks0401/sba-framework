## 1. 基本方針

### 1.1 無料運用優先原則
SBAは、無料運用を最大化し、有料APIの利用は最終手段とする。[file:1]  
ローカル完結できる処理は、Ollama、FAISS、Neo4j、Whisper、Browser-Use、yt-dlp などの無料構成を優先する。[file:1]

### 1.2 コスト発生回避の考え方
外部APIは「ローカルで代替できない処理」または「時間短縮効果が高い処理」に限定して使用する。[file:1]  
同じ目的を無料ソースで達成できる場合は、無料ソースを必ず優先する。[file:1]

### 1.3 停止条件の考え方
各APIには、警告閾値・抑制閾値・停止閾値を設ける。  
停止閾値に達した場合、その日の新規呼び出しを停止し、代替ルートまたは翌日再開に切り替える。[file:1]

## 2. 管理対象一覧

### 2.1 完全無料層
以下は原則コスト0円で運用できるため、常時優先利用対象とする。[file:1]

- Ollama（Phi-4:14B / Qwen2.5-Coder:7B / Llama3.1:8B）: ローカルLLM推論。[file:1]
- FAISS: Vector DB。[file:1]
- Neo4j Community Edition: Graph DB。[file:1]
- Whisper: 音声→テキスト変換。[file:1]
- BeautifulSoup4 + Playwright + Jina Reader: Web取得。[file:1]
- Browser-Use: 自律ブラウザ操作。[file:1]
- yt-dlp: 動画・字幕取得。[file:1]
- arXiv / Wikipedia / OpenAlex / Project Gutenberg: 学術・百科・文書取得。[file:1]

### 2.2 無料枠活用層
以下は無料枠があるため、上限超過に厳重注意で運用する。[file:1]

- Gemini 2.5 Flash: 15 req/min・1,500 req/day・1M tokens/min。[file:1]
- YouTube Data API v3: 10,000 units/day。[file:1]
- NewsAPI: 100 req/day（開発者プラン）。[file:1]
- Hugging Face Inference API: 月30,000 req。[file:1]
- GitHub API: 認証あり5,000 req/hour、匿名60 req/hour。[file:1]
- Stack Overflow API: 10,000 req/day。[file:1]

### 2.3 管理対象外
ローカルCPU/RAM/SSD使用量は本書の主対象ではないが、VRAM競合や長時間実行に伴う停止判断は関連資料で連携管理する。  
VRAM制御の詳細は「推論エンジン・VRAM運用設定書」で定義する。[file:1]

## 3. API別運用ルール

### 3.1 Gemini 2.5 Flash

#### 3.1.1 用途
Gemini 2.5 Flash は、大量テキストの要約・構造化・Tier1補完処理に使用する。[file:1]

#### 3.1.2 利用条件
以下の場合のみGemini利用を許可する。[file:1]

1. 処理対象token数が8,000を超える場合。[file:1]
2. Tier1の待機時間が10秒を超える場合。[file:1]
3. 長文PDFや動画字幕など、Tier1よりTier2の方が明確に効率的な場合。[file:1]

#### 3.1.3 制限値
- 15 req/min [file:1]
- 1,500 req/day [file:1]
- 1M tokens/min [file:1]

#### 3.1.4 停止条件
Gemini残枠が100を切ったら、その日の新規外部API呼び出しを停止する。[file:1]  
停止後は、Tier1分割処理または翌日再実行へ切り替える。[file:1]

---

### 3.2 YouTube Data API v3

#### 3.2.1 用途
YouTube Data API v3 は、動画メタデータ、字幕一覧、チャンネル情報取得に使用する。[file:1]

#### 3.2.2 運用方針
字幕そのものの取得は yt-dlp を優先し、APIは補助用途に限定する。[file:1]  
動画検索や候補絞り込みが必要なときだけ使う。[file:1]

#### 3.2.3 制限値
- 10,000 units/day [file:1]

#### 3.2.4 停止条件
日次使用量が上限の80%を超えたら警告ログを出し、95%を超えたら非必須リクエストを停止する。  
100%到達時は当日中の新規呼び出しを全面停止し、yt-dlp中心の代替ルートに切り替える。[file:1]

---

### 3.3 NewsAPI

#### 3.3.1 用途
NewsAPI は、時事性が必要なBrainに対するニュース取得に限定利用する。[file:1]

#### 3.3.2 制限値
- 100 req/day [file:1]

#### 3.3.3 運用方針
NewsAPI は優先度を低めに設定し、他ソースで代替可能な場合は使用しない。[file:1]

#### 3.3.4 停止条件
日次使用量が70%を超えたら低優先度ジョブでの使用を停止する。  
90%を超えたら緊急用途以外を停止し、100%到達で全面停止とする。

---

### 3.4 Hugging Face Inference API

#### 3.4.1 用途
Hugging Face Inference API は、モデル・データセット・ペーパー取得などの補助用途で利用する。[file:1]

#### 3.4.2 制限値
- 月30,000 req [file:1]

#### 3.4.3 停止条件
月次使用量が80%を超えたら新規大量取得ジョブを停止する。  
95%を超えたら手動許可ジョブのみ実行可能とし、100%到達で翌月まで停止する。

---

### 3.5 GitHub API

#### 3.5.1 用途
GitHub API は、README、コード、Issue、Wiki取得に利用する。[file:1]

#### 3.5.2 制限値
- 認証あり: 5,000 req/hour [file:1]
- 匿名: 60 req/hour [file:1]

#### 3.5.3 運用方針
GitHub API は認証あり利用を前提とする。[file:1]  
レートヘッダを毎回記録し、残量低下時はクロール速度を自動で落とす。

#### 3.5.4 停止条件
残reqが10%未満になったら、高頻度クロールを停止し、README優先モードへ移行する。  
0に達した場合は、その時間帯の処理を停止し、次の時間窓まで待機する。

---

### 3.6 Stack Overflow API

#### 3.6.1 用途
Stack Overflow API は、Q&A形式の実務的解決パターン取得に使う。[file:1]

#### 3.6.2 制限値
- 10,000 req/day [file:1]

#### 3.6.3 停止条件
80%超で大量巡回を停止、95%超で検索系だけに限定、100%で全面停止とする。  
停止後は、既存取得データとGitHub・公式ドキュメントを優先する。

## 4. 優先順位と代替ルート

### 4.1 検索・Web取得
検索は DuckDuckGo を優先し、有料検索APIは原則使わない。[file:1]  
Web本文取得は Jina Reader を先に試し、失敗時のみ Browser-Use / Playwright にフォールバックする。[file:1]

### 4.2 動画・字幕
YouTube字幕は yt-dlp を優先し、YouTube Data API は補助用途とする。[file:1]  
字幕がない場合のみ Whisper を使う。[file:1]

### 4.3 長文処理
長文要約は、まずローカル処理を検討し、token数8,000超や待機時間超過時のみGeminiへ逃がす。[file:1]

### 4.4 学術・統計
論文は arXiv / OpenAlex、統計は e-Stat / World Bank を優先し、課金系ソースは採用しない。[file:1]

## 5. カウンタ・停止制御実装方針

### 5.1 カウンタ管理
全APIのデイリー使用量は SQLite でカウントする。[file:1]  
少なくとも API名、日付、使用回数、使用token数またはunit数、最終更新日時を保持する。

### 5.2 スロットリング
上限前に自動スロットリングを実装し、急な上限到達を防ぐ。[file:1]  
スロットリングは、待機時間追加、ジョブ優先度低下、代替ソース切替で実現する。

### 5.3 リセット処理
日次制限APIは、日付変更時にカウンタをリセットする。  
月次制限APIは、月初にカウンタをリセットする。

### 5.4 判定段階
各APIについて、最低でも以下の3段階を持つ。  
- WARNING: 警告のみ  
- THROTTLE: 非必須処理停止  
- STOP: 当日または当月の新規呼び出し停止

## 6. 運用ルール

### 6.1 非必須ジョブの定義
以下は非必須ジョブとして、制限逼迫時に真っ先に止める。  
- 低信頼ソースの追加巡回  
- ニュース再取得  
- 重複可能性の高い広域検索  
- 優先度の低い動画メタデータ取得

### 6.2 必須ジョブの定義
以下は必須ジョブとして、停止直前まで維持対象とする。  
- 弱点SubSkillの補完学習  
- 重量実験前の最低限知識補完  
- エクスポート対象Brainの品質確保処理

### 6.3 手動解除
停止状態を手動解除する場合は、課金発生リスクを明示したうえで明示承認を必要とする。  
初期実装では、手動解除は運用者専用設定ファイルまたはCLIオプション経由に限定する。

## 7. 監査・ログ

### 7.1 記録対象
- API名
- 呼び出し日時
- Brain名
- SubSkill
- 呼び出し理由
- 消費量（req / unit / token）
- 実行結果
- スロットリング判定結果

### 7.2 可視化
日次・月次で API別利用量、停止回数、代替ルート発動回数を集計する。  
特に Gemini、YouTube Data API、NewsAPI は、停止条件に直結するのでダッシュボード化対象とする。[file:1]

## 8. 例外時の扱い

### 8.1 API障害時
API自体が落ちている場合は、残枠があっても停止扱いとし、リトライ回数上限を超えたら代替ソースへ切り替える。  
外部障害は「利用量不足」と区別してログ記録する。

### 8.2 代替不能時
代替ソースが存在しない場合、その学習ジョブは延期し、翌日再開候補キューへ積む。  
課金許可がない限り、有料代替への自動切替は行わない。

## 9. 不変事項と調整可能事項

### 9.1 不変事項
- 無料運用最大化を最優先とする。[file:1]
- Gemini残枠100未満で当日新規呼び出し停止とする。[file:1]
- DuckDuckGoで代替できる場合は有料検索を使わない。[file:1]
- YouTube字幕は yt-dlp を優先し、APIは補助のみとする。[file:1]
- 全API使用量はSQLiteでカウントし、上限前に自動スロットリングする。[file:1]

### 9.2 調整可能事項
- Warning / Throttle / Stop の閾値
- 非必須ジョブの定義
- リトライ回数
- 代替ルートの優先順位
- ダッシュボード集計粒度
