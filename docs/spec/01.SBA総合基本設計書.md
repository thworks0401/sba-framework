1. プロジェクト概要
1.1 プロジェクト名称
SpecializedBrainAgent（SBA）Framework
1.2 コンセプト（一行定義）
「育成専用エージェントが自律的に学習・検索・実験を繰り返し、1ドメインの脳みそを100Lvまで育て上げるBrain Factory。育てた脳みそは保管し、別ドメインはまっさらな脳みそで差し替えて再育成。育成済みの脳みそはエクスポートして別途作成する実務ツールに渡す。」
1.3 プロジェクトの目的
SBAは、特定ドメインに特化した高品質な知識データベース（Brain）を自律的に育成・製造・保管し、実務ツールで利用可能な形式でエクスポートすることを唯一の目的とする。
具体的な達成目標:
•	人間の介入を最小化した完全自律学習システムの確立
•	ドメイン別に最適化されたSubSkill体系による知識整理
•	実験を通じた体験的学習による信頼性の高い知識蓄積
•	Brain Hot-Swap機構による複数ドメインの効率的な育成運用
•	育成済みBrainの実務ツールへのシームレスな引き渡し
1.4 プロジェクトの非目的（絶対的制約）
SBAは以下の機能を一切持たない。この制約は設計上の絶対的前提であり、変更不可とする。
 	実務処理の実行（文書作成、コード実行、業務判断、返信送付等）
 	実務ツールとの直接通信・連携・API呼び出し
 	ユーザーへの実務アウトプット提供
 	複数エージェントの同時並列動作（1体に全リソース集中が原則）
 	リアルタイム応答・チャットボット機能
 	外部システムへのプッシュ通知・自動送信
補足: SBAと実務ツールの連携は、SBAが生成するエクスポートパッケージを介した間接的な連携のみとする。連携の仕組み自体は別プロジェクトで設計する。
1.5 想定利用シーン
1.	開発初期: Python開発Brainを育成し、コーディング・デバッグ・設計知識を蓄積
2.	事業拡大期: 法人営業Brainを育成し、営業手法・提案パターンを体系化
3.	確定申告期: 青色申告Brainを装着し、最新の税制・控除情報で自己更新
4.	新規事業: 育成済みPython開発Brainをエクスポートし、開発支援ツールに組み込み
5.	複数ドメイン運用: 平日はPython開発、週末は不動産投資Brainに差し替えて並行育成

設計原則:
•	Agent Coreは全Brainで共通利用し、一切変更しない
•	Brainは完全に独立したデータパッケージとして保管される
•	同時に装着できるBrainは常に1つのみ（排他制御）
•	Brain切替時はAgent Coreを再起動してコンテキストをクリア
2.2 Brain の4層内部構造
1つのBrainは以下の4層で構成される。各層は独立したデータベースまたはファイルとして管理される。
Layer	役割	実装技術
Layer 1: Knowledge Base	学習済み知識の格納・検索	FAISS + Neo4j
Layer 2: Experiment Log	自己実験の成功・失敗体験記録	SQLite
Layer 3: Self-Evaluation	SubSkillごとの自己評価スコア	JSON + SQLite
Layer 4: Learning Timeline	学習履歴・ソース・鮮度管理	SQLite

Table 1: Brain内部の4層構造
補足: 各層の詳細仕様は「Brain内部構造設定書」で定義する。
2.3 主要処理フロー
SBAの主要な処理フローは以下の5つのエンジンで構成される。
1.	自律学習エンジン: 知識ギャップ検出 → リソース探索 → 取得 → クリーニング → 仕分け → 格納
2.	自己実験エンジン: 仮説生成 → 実験設計 → 実行 → 評価 → 知識更新
3.	SubSkill仕分けエンジン: 取得知識のドメイン内分類・重複判定・格納先決定
4.	推論エンジン切替システム: Tier1/Tier2/Tier3の動的切替・VRAM排他制御
5.	Brain管理エンジン: Hot-Swap（保存/ロード/差し替え/エクスポート）
補足: 各エンジンの詳細仕様は個別の設定資料で定義する。

3. 主要コンポーネント定義
3.1 Agent Core（エージェント本体）
責務:
•	装着されたBrainの知識を活用した推論実行
•	入力クエリのSubSkillへのルーティング
•	3段階の推論エンジン（Tier1/Tier2/Tier3）の切替制御
•	自律学習・自己実験の定期実行制御
•	VRAM使用量の監視と排他制御
技術スタック:
•	推論エンジン: Ollama（Phi-4:14B, Qwen2.5-Coder:7B, Llama3.1:8B）
•	外部API: Google Gemini 2.5 Flash（無料枠利用）
•	スケジューラ: APScheduler 3.x（BackgroundScheduler + SQLiteJobStore）
•	言語: Python 3.11（venv固定）
•	OS: Windows 11 / PowerShell
補足: Agent Coreは全Brain共通で利用するため、Brain固有の設定を一切持たない設計とする。
3.2 Brain（脳みそパッケージ）
定義: 特定ドメインの知識・実験記録・評価スコア・学習履歴を含む、完全に独立したデータパッケージ。
Brain 1個の構成要素:
•	knowledge_graph.db: Neo4jダンプファイル（知識グラフ）
•	vector_index.faiss: FAISSインデックス（ベクトル検索用）
•	experiment_log.db: SQLiteデータベース（実験記録）
•	learning_timeline.db: SQLiteデータベース（学習履歴）
•	self_eval.json: SubSkillごとの評価スコア（JSON形式）
•	subskill_manifest.json: SubSkill構造定義（JSON形式）
•	metadata.json: ドメイン・バージョン・Lv・最終更新日時
Brainのライフサイクル:
1.	生成: Blank Brain Templateからコピー作成
2.	育成: Agent Coreに装着して自律学習・自己実験を実行
3.	保管: Brain Bankにバージョン管理して永続保存
4.	再装着: 保管済みBrainを再ロードして育成継続
5.	エクスポート: 実務ツール用のパッケージとして出力
補足: Brainの詳細構造は「Brain内部構造設定書」、Hot-Swap仕様は「Brain Hot-Swap設定書」で定義する。
3.3 Brain Bank（脳みそ保管庫）
定義: 育成済みBrainを成長状態のまま永続保管するディレクトリベースのリポジトリ。
ディレクトリ構造:
C:/SBA/brain_bank/
├── _blank_template/ ← まっさらな脳みそのマスターコピー（変更禁止）
├── Python開発_v1.4/ ← 育成済みBrain（保管状態）
├── 法人営業_v0.8/
├── 青色申告_v2.1/
└── [active]/ ← 現在装着・育成中のBrain
管理方針:
•	各Brainはバージョン管理される（例: Python開発_v1.4）
•	_blank_template は常に初期状態を保持し、変更禁止
•	[active] は現在育成中のBrainのシンボリックリンクまたはコピー
•	Brain保管時は自動的にバージョン番号を付与
•	同一ドメインの複数バージョンを並行保管可能
補足: Brain Bankの詳細仕様は「Brain Package・保存形式設定書」で定義する。
3.4 SubSkill（サブスキル体系）
定義: 1つのBrain内で知識を用途別に分類するための中分類タグ。
SubSkillの役割:
•	取得した知識を用途別に自動仕分け
•	知識検索時の精度向上（SubSkill指定検索）
•	弱点領域の自動検出と優先学習
•	SubSkillごとの信頼スコア管理
•	育成Lv評価の基礎単位
SubSkill例（Python開発Brain）:
•	SubSkill: 設計（アーキテクチャ・要件定義・設計パターン）
•	SubSkill: 実装（コーディング規約・実装手法・ライブラリ活用）
•	SubSkill: デバッグ（エラーパターン・スタックトレース解析）
•	SubSkill: 修正（リファクタリング・パフォーマンス最適化）
•	SubSkill: テスト（ユニットテスト・結合テスト・カバレッジ管理）
•	SubSkill: レビュー（コードレビュー基準・品質指標）
•	SubSkill: ドキュメント（README・docstring・API仕様書作成）
補足: SubSkillの命名規則、自動仕分けアルゴリズムは「SubSkill体系設定書」で定義する。
3.5 推論エンジン（3段階構成）
SBAは用途に応じて3種類のLLMを動的に切り替える。
Tier	モデル	VRAM	用途
Tier1	Phi-4:14B	8.0GB	深い推論・SubSkill仕分け・自己評価
Tier2	Gemini 2.5 Flash	0GB	大量テキスト要約・PDF処理
Tier3	Qwen2.5-Coder:7B	5.0GB	Tech系のコード生成・検証

Table 2: 推論エンジンの3段階構成
切替ロジック:
1.	タスクがコード生成・検証 AND Brain=Tech系 → Tier3
2.	処理token数 > 8,000 OR Tier1待機超過10秒 → Tier2
3.	上記以外の全タスク → Tier1（デフォルト）
VRAM排他制御:
•	Tier1とTier3は同時起動禁止（VRAM競合のため）
•	Whisper使用時はOllamaモデルを一時アンロード
•	VRAM使用量を常時監視し、OOM発生前に自動切替
補足: エンジン切替の詳細条件とVRAM制御は「推論エンジン・VRAM運用設定書」で定義する。
3.6 学習ソース（データ収集元）
SBAは以下のソースから自律的にデータを収集する。
完全無料ソース:
•	Web検索: DuckDuckGo Search, Jina Reader API, Browser-Use
•	学術文献: arXiv API, OpenAlex API, Wikipedia API
•	コード情報: GitHub API（5,000 req/h）, Stack Overflow API（10,000 req/day）
•	動画・音声: yt-dlp（字幕取得）, Whisper（音声変換）
•	構造化データ: Wikidata API, DBpedia SPARQL, e-Stat API
無料枠ありソース（超過注意）:
•	Gemini 2.5 Flash API: 15 req/min, 1,500 req/day
•	YouTube Data API v3: 10,000 units/day
•	NewsAPI: 100 req/day
•	Hugging Face Inference API: 30,000 req/month
補足: 各ソースの優先度、信頼スコア、利用制限は「学習ソース・収集方針設定書」で定義する。

4. Brain Hot-Swap機構
4.1 Hot-Swapの定義
Hot-Swap: Agent Core（推論エンジン）を固定したまま、装着されているBrain（知識DB）を別のBrainに差し替える機構。
設計思想:
•	エージェント本体は1体のみ・常時稼働・変更しない
•	差し替えるのはBrain = 知識DB + SubSkillモジュール のセット
•	複数エージェントを並行動作させず、1体に全リソースを集中
•	Brainごとに独立した育成状態を保持
4.2 Blank Brain Template（まっさらな脳みそ）
定義: 学習データ0・知識0の初期状態のBrainパッケージ。新ドメインの育成開始時は必ずこのテンプレートのコピーから生成する。
構成:
•	knowledge_graph.db: Neo4jダンプ（空）
•	vector_index.faiss: FAISSインデックス（空）
•	experiment_log.db: SQLite（空）
•	learning_timeline.db: SQLite（空）
•	self_eval.json: 全SubSkillスコア初期値0.0
•	subskill_manifest.json: SubSkill構造定義（空配列）
•	metadata.json: {domain: null, version: "0.0", level: 0}
運用ルール:
•	テンプレート本体は変更・上書き禁止（マスターコピーとして永続保管）
•	新Brainの生成は常にBlank Brain Templateのコピーから開始
•	テンプレートの破損を防ぐため、読み取り専用属性を推奨
4.3 育成・保管・再利用の流れ
1.	Step1: Blank Templateを複製 → 「Python開発Brain」として生成
2.	Step2: Python開発Brainを育成（学習・実験・自己整理）
3.	Step3: ある時点でBrain Bankに保管（成長状態のまま）
4.	Step4: 新たにBlank Templateを複製 → 「法人営業Brain」として生成
5.	Step5: 法人営業Brainを育成
6.	Step6: 必要な時にPython開発Brainを再装着 → 育成続行または即エクスポート
補足: Hot-Swapの詳細仕様（コマンド、内部処理フロー）は「Brain Hot-Swap設定書」で定義する。

5. 育成Lv定義（全知全能への到達基準）
5.1 Lv体系の概要
SBAはBrainの育成度を3段階のLvで管理する。Lvは全SubSkillの平均スコアと定性的な能力で判定される。
Lv	名称	達成基準
Lv.1	Practitioner（実務家）	全SubSkill平均スコア80%以上
Lv.2	Expert（専門家）	全SubSkill平均スコア95%以上 + 説明能力
Lv.3	Domain Sovereign（領域主権者）	平均スコア99%以上 + 自律補完ループ稼働

Table 3: 育成Lv定義
5.2 各Lvの詳細定義
Lv.1 Practitioner（実務家レベル）
•	定義: 現役プロと同等の知識・判断力を持つ
•	達成基準: 全SubSkillにおいて自己採点の平均スコアが80%以上
•	特徴: 標準的なタスクを正確に遂行できる。エッジケースは未対応可。
Lv.2 Expert（専門家レベル）
•	定義: その分野のトップ5%相当。エッジケース・例外・最新動向まで網羅。
•	達成基準: 全SubSkillにおいて自己採点の平均スコアが95%以上 + 他者への説明・設計ができる（説明生成テストで合格）
•	特徴: 教える・設計できる・新人への指導が可能なレベル。
Lv.3 Domain Sovereign（領域主権者レベル）
•	定義: その分野において「右に出る者がいない」状態。新しい問いに対して自律的に仮説を生成・検証できる。自分の知識不足を自覚し、自律的に補完行動をとれる。
•	達成基準: 全SubSkillの平均スコア99%以上を維持 + 知識不足フラグ発火 → 自律補完 → 再評価 のループが稼働している状態
•	特徴: 完全到達は理論上不可能なため「漸近目標」として扱う。自律補完ループが常時稼働している状態をLv.3と定義する。
補足: Lv評価の自動判定ロジック、スコア計算方法は「成長評価・Lv定義設定書」で定義する。

6. 主要処理エンジン概要
6.1 自律学習エンジン
役割: 人間の介入なしで知識ギャップを検出し、自律的にリソースを探索・取得・整理・格納する。
処理フロー:
1.	知識ギャップ検出（Self-Evaluationから弱点SubSkillを特定）
2.	学習リソース自動探索（優先度に従いソースを選択）
3.	データ取得・クリーニング・構造化
4.	SubSkill自動仕分け
5.	知識統合・再構造化
6.	育成Lv自己評価・次学習優先度決定
補足: 詳細仕様は「自律学習ループ設定書」で定義する。
6.2 自己実験エンジン
役割: 知識を得るだけでなく、実際に試して体験から学ぶ。定期的に自律実行し、成功・失敗の全体験をExperiment Logに永続保管する。
実験種別:
•	種別A: 知識確認実験（問題生成→自己回答→正誤判定）
•	種別B: 推論実験（論理問題生成→多段推論→矛盾チェック）
•	種別C: コード実験（コード生成→サンドボックス実行→結果検証）※Tech系のみ
•	種別D: シミュレーション実験（仮想ビジネスケース生成→判断→評価）
実験スケジューラ:
•	軽量実験（種別A）: 1時間ごと、5〜15分、3〜5問
•	中量実験（種別C, D）: 6時間ごと、30〜60分、1〜2ケース
•	重量実験（総合評価）: 24時間ごと、2〜4時間、全SubSkill評価
補足: 詳細仕様は「自己実験エンジン設定書」で定義する。
6.3 SubSkill自動仕分けエンジン
役割: 取得した知識を適切なSubSkillに自動分類し、重複を検出・統合する。
処理手順:
1.	取得テキストをTier1（Phi-4:14B）に入力
2.	「このテキストは[BrainName]のどのSubSkillに該当するか。該当SubSkillを全て列挙し、主SubSkillを1つ選べ」と自己プロンプト
3.	主SubSkillに格納・副SubSkillには参照リンクのみ生成（重複格納しない）
4.	既存の類似知識との類似度をFAISSで検索（コサイン類似度 > 0.92 で重複判定）
5.	重複の場合は新情報で既存を更新 or 補完（古い情報フラグ付与）
6.	Neo4j に「知識ノード」として登録・SubSkillノードとEDGEで接続
補足: 詳細仕様は「SubSkill体系設定書」で定義する。
6.4 Brain管理エンジン
役割: Brain Bankへの保存、ロード、差し替え、エクスポートを管理する。
主要コマンド:
•	sba brain swap: 現在のBrainを保管して別Brainに差し替え
•	sba brain load: 保管済みBrainを再装着
•	sba brain list: Brain Bank一覧表示
•	sba brain export: 育成済みBrainを実務ツール用にエクスポート
補足: 詳細仕様は「Brain Hot-Swap設定書」「Brain Package・保存形式設定書」で定義する。

9. エクスポート仕様（実務ツールへの引き渡し）
9.1 エクスポートの定義
エクスポート: 育成済みBrainを実務ツールで利用可能な形式でパッケージ化して出力すること。
設計原則:
•	SBAはエクスポートまでが責務範囲
•	エクスポート後の利用はSBA完全不関与
•	実務ツールとの連携の仕組みは別プロジェクトで設計する
9.2 エクスポート出力物
C:/exports/Python開発_v1.4/
├── knowledge_graph.db ← Neo4j dump（読み取り専用）
├── vector_index.faiss ← FAISSインデックス（読み取り専用）
├── experiment_log.db ← 成功/失敗体験DB（読み取り専用）
├── subskill_manifest.json ← SubSkill構造・タグマップ
├── metadata.json ← ドメイン・バージョン・達成Lv・エクスポート日時
└── brain_api.py ← 実務ツール用APIラッパー（SBA依存なし）
9.3 brain_api.py の役割
定義: 実務ツールがエクスポートされたBrainを利用するためのPython APIラッパー。SBAに依存せず、スタンドアロンで動作する。
主要メソッド:
•	brain.query(question, subskill, top_k): SubSkill指定クエリ
•	brain.list_subskill(subskill_name): 特定SubSkillの全知識一覧取得
•	brain.info(): Brainのメタ情報取得
•	brain.eval_scores(): SubSkillごとの信頼スコア一覧取得
補足: エクスポート仕様、brain_api.pyの詳細設計は「エクスポート・外部利用仕様書」で定義する。
