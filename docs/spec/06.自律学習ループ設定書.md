## 1. 自律学習ループの概要

### 1.1 コンセプト
SBAは人間の介入なしで、自ら弱点を検知し、学習リソースを探索し、データを取得・整理し、知識を統合し、自己評価を更新する学習サイクルを常時自律実行する。[file:1]

### 1.2 メインループ構成
メインループは以下のStepで構成される。[file:1]

1. Step1: 知識ギャップ検出
2. Step2: 学習リソース自動探索
3. Step3: データ取得・クリーニング・構造化
4. Step4: 自己実験（Self-Experimentation Engineへ委譲）
5. Step5: 知識統合・再構造化
6. Step6: 育成Lv自己評価・次学習優先度決定

### 1.3 実行形態
- 実行単位: 装着中の1 Brainごと
- 実行タイミング: 常時バックグラウンド / スケジューラ制御
- 優先順位: 自己実験、APIレート制限、VRAM使用状況と連携

## 3. Step1: 知識ギャップ検出

### 3.1 目的
BrainのどのSubSkillが最も弱いか（知識密度が低いか）を検出し、学習優先対象を決定する。[file:1]

### 3.2 入力
- Self-Evaluation（Layer3）
  - SubSkillごとの知識密度スコア（0.0〜1.0）
  - 弱点フラグ
  - 補完学習優先度キュー[file:1]

### 3.3 出力
- 優先学習対象SubSkill（1件以上）
- 学習クエリ生成用の内部プロンプト

### 3.4 処理概要
1. 全SubSkillの知識密度スコアを取得する。[file:1]
2. 最小スコアのSubSkillを優先学習対象として選定する（閾値・優先度キューを考慮）。[file:1]
3. 「何を知らないか」を言語化した学習クエリをTier1（Phi-4:14B）で自動生成する。[file:1]

### 3.5 補足
- 直近で集中的に学習したSubSkillは一定期間クールダウンしてもよい（飽和防止）。
- 将来的にはSubSkill間の依存関係グラフを考慮した優先度計算も拡張対象。

---

## 4. Step2: 学習リソース自動探索

### 4.1 目的
Step1で選定したSubSkillについて、外部・内部のデータ取得ソースから最適な学習リソースを自律的に探索する。[file:1]

### 4.2 入力
- 優先学習対象SubSkill
- 学習クエリ
- データ取得ソース一覧と優先度
- API使用量・レート制限状況[file:1]

### 4.3 出力
- 取得候補リソースのリスト（URL・APIクエリ・ローカルパス 等）
- 各リソースに対する初期信頼スコア

### 4.4 信頼スコア付与ルール（初期値）
- 学術論文（arXiv, OpenAlex 等）: 0.95[file:1]
- 公式ドキュメント（docs, MDN 等）: 0.90[file:1]
- 専門サイト: 0.80[file:1]
- 一般ブログ: 0.65[file:1]
- SNS・掲示板: 0.50[file:1]

信頼スコア0.60未満のデータには「要確認」フラグを付与する。[file:1]

### 4.5 データソース選択ロジック（概要）
1. SubSkillの性質に応じてソース優先度を切り替える（例: Tech系ならGitHub/Stack Overflow優先）。[file:1]
2. 無料枠・レート制限を考慮し、Gemini / NewsAPI 等の有償系APIを慎重に使用する。[file:1]
3. 同一URLの再取得や明らかな重複はLearning Timelineを参照して避ける。[file:1]

---

## 5. Step3: データ取得・クリーニング・構造化

### 5.1 目的
選定したリソースから生データを取得し、ノイズ除去・正規化・分割・重複排除・SubSkill仕分けを行い、Knowledge Baseに格納できる形にする。[file:1]

### 5.2 入力
- リソースリスト（URL、APIクエリ、ファイルパスなど）
- 初期信頼スコア
- 対象SubSkill

### 5.3 出力
- Knowledge Baseへ格納された知識ノード群
- Learning Timelineへの履歴レコード
- 必要に応じた「要確認」「古い情報」等のフラグ

### 5.4 処理ステップ概要
1. データ取得  
   - Web: DuckDuckGo → Jina Reader → HTML/Markdown取得[file:1]  
   - 動的サイト: Browser-Use / Playwright[file:1]  
   - PDF: arXiv / PDFMiner / pypdf[file:1]  
   - 動画: yt-dlp（字幕） or Whisper（音声→テキスト）[file:1]  
   - コード: GitHub API / Stack Overflow API / PyPI JSON API 等[file:1]

2. クリーニング
   - 広告・ナビゲーション・不要HTMLの除去
   - テキスト正規化
   - 文章単位・チャンク単位での分割[file:1]

3. 重複チェック
   - ハッシュ値による完全重複排除（Learning Timelineで管理）[file:1]
   - FAISSによる意味的重複判定（コサイン類似度 > 0.92 で重複とみなす）[file:1]

4. SubSkill自動仕分け
   - Tier1（Phi-4:14B）へ「このテキストはどのSubSkillか」を自己プロンプトして主SubSkill + 副SubSkillを決定。[file:1]
   - 主SubSkillに格納、副SubSkillには参照リンクのみ生成（重複格納なし）。[file:1]

5. 格納
   - Knowledge Base（Layer1）へ保存（FAISS + Neo4j）。[file:1]
   - Learning Timeline（Layer4）へソース種別・URL・ハッシュ・SubSkill・鮮度スコアを保存。[file:1]

## 6. Step4: 自己実験への委譲

### 6.1 目的
学習した知識を「実際に試す」ことで、理解の深さや誤りを浮かび上がらせる。[file:1]

### 6.2 入力
- 弱点SubSkill（Step1）
- 新規学習知識（Step3）
- Self-Evaluationの現状スコア

### 6.3 出力
- Experiment Logへの実験記録
- 知識の更新候補
- Self-Evaluation更新用のスコア変化

### 6.4 処理の位置付け
- 自己実験そのものの詳細は「自己実験エンジン設定書」で定義し、本書では「Step4でSelf-Experimentation Engineへ制御を渡す」ことを明示する。[file:1]

## 7. Step5: 知識統合・再構造化

### 7.1 目的
新しく得た知識と既存のKnowledge Baseとの間の矛盾や重複を検出し、整合性を取ったうえで知識グラフを再構造化する。[file:1]

### 7.2 入力
- Knowledge Baseの既存ノード
- 新規追加候補ノード
- Experiment Logの結果
- Learning Timeline（鮮度・ソース種別）

### 7.3 出力
- 更新されたKnowledge Base
- deprecated / 要確認 等のフラグ付きノード
- 矛盾フラグ付きノードペア

### 7.4 処理概要
1. 矛盾検出
   - Neo4j上の関係性や、明示的に矛盾しうる知識ペアを抽出。
2. 矛盾時の処理
   - 両方に「矛盾フラグ」を付与。[file:1]
   - 新旧の信頼スコア（+ソース種別）を比較し、スコアの高い方を主系とする。[file:1]
   - 旧情報を「deprecated」へ、必要に応じて「古い情報フラグ」を付与。[file:1]
   - 判断不能な場合は「要人間確認」フラグを付与し、Slack通知など外部連携へ。 ※通知詳細は別資料[file:1]

3. 知識グラフの最適化
   - Neo4jのEDGEを再構築し、冗長なリンクや不整合を整理。[file:1]

## 8. Step6: 育成Lv自己評価・次学習優先度決定

### 8.1 目的
現時点のBrainの育成状況（Lv.1 / Lv.2 / Lv.3）を自己評価し、次ループの学習優先度を再計算する。[file:1]

### 8.2 入力
- Knowledge Base（SubSkillごとの知識量・多様性）
- Experiment Log（成功/失敗履歴）
- Learning Timeline（鮮度・更新頻度）
- 現在のSelf-Evaluationスコア

### 8.3 出力
- 更新されたSelf-Evaluation（SubSkillスコア・弱点フラグ・Lv）
- 次ループにおける優先学習SubSkillの再計算結果

### 8.4 評価ロジック概要
1. ランダム問題生成・自己回答・自己採点
   - Tier1（Phi-4:14B）にSubSkillごとの問題生成と自己回答・自己採点をさせる。[file:1]
2. スコア算出
   - SubSkillごとのスコアを0.0〜1.0で算出し、平均値・分布に基づきLv.1/2/3を判定。[file:1]
   - Lv.1: 平均80%以上（Practitioner）[file:1]
   - Lv.2: 平均95%以上 + 説明生成テスト合格（Expert）[file:1]
   - Lv.3: 平均99%以上 + 自律補完ループ常時稼働状態（Domain Sovereign）[file:1]

3. 次学習優先度の決定
   - スコアの低いSubSkillから優先度キューを更新。
   - 弱点フラグ（0.6以下）を基準に補完対象を列挙。[file:1]

## 9. ループ制御とスケジューリング

### 9.1 ループの粒度
- 1ループあたり: Step1〜6までを1サイクルとし、Brainごとに逐次実行する。
- Loop間インターバルや上限時間は設定可能とする。

### 9.2 自己実験とのスケジューラ連携
自己実験エンジンのスケジューラ（軽量 / 中量 / 重量実験）との整合を取る必要がある。[file:1]

- 軽量実験: 1時間ごと（5〜15分、3〜5問）[file:1]
- 中量実験: 6時間ごと（30〜60分、1〜2ケース）[file:1]
- 重量実験: 24時間ごと（2〜4時間、全SubSkill総合評価）[file:1]

メインループは、これらの実験スケジュールを考慮してStep4〜6の実行タイミングを調整する。

### 9.3 外部APIレート制限との統合
- Gemini、YouTube Data API、NewsAPI等は日次上限があるため、SQLiteベースのカウンタで日次使用量を監視する。[file:1]
- 上限に近づいた場合、Step2で外部API依存のリソースを抑制し、ローカル/無料ソース優先に切り替える。[file:1]

## 10. エンジン切替との関係

### 10.1 タスク種別によるTier選択
- コード生成・検証かつBrainがTech系 → Tier3（Qwen2.5-Coder）[file:1]
- 処理token数 > 8,000 またはTier1待機10秒超過 → Tier2（Gemini）[file:1]
- 上記以外 → Tier1（Phi-4:14B）[file:1]

### 10.2 学習ループへの影響
- Step1〜3: 主にTier1 / Tier2を利用（長文処理時はTier2）。[file:1]
- Step4: 自己実験の種別C（コード実験）ではTier3 + subprocessを利用。[file:1]
- Step5〜6: 主にTier1を利用（矛盾検出、評価・採点）。[file:1]

## 11. 例外処理と人間介入ポイント

### 11.1 自動判断が難しいケース
- 矛盾解消が信頼スコアだけでは決めきれない場合
- 法律・税務など、人間判断が必須なドメインの一部トピック
- APIエラー・ネットワークエラーが頻発し学習が進まない状態

### 11.2 人間介入のタイミング
- Step5の「要人間確認」フラグが立ったノード群に対し、Slack通知などでアラートを出す。[file:1]
- Self-Evaluationで異常値（急激なスコア変動等）が出た際にダッシュボード表示・確認依頼を行う。

## 12. 設計上の不変事項と拡張可能事項

### 12.1 不変事項
- 自律学習ループは「知識ギャップ検出 → 学習 → 統合 → 自己評価」の流れを持つ。[file:1]
- Step1〜6の存在と役割は維持する。
- SubSkill単位の優先度管理を行う。

### 12.2 拡張可能事項
- 学習リソース選択アルゴリズム（Rankingロジック）の高度化
- 矛盾検出ロジックの強化（パターンマッチ・ルールベースの追加）
- Self-Evaluationのスコアリングモデルの高度化（統計モデル・RL等）
