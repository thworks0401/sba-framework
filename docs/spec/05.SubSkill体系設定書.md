## 1. Brain内部4層モデル概要

### 1.1 4層モデルの全体像
1つのBrainは、以下の4つの論理層で構成される。[file:1]

1. Layer 1: Knowledge Base（知識層）
2. Layer 2: Experiment Log（実験体験層）
3. Layer 3: Self-Evaluation（自己評価層）
4. Layer 4: Learning Timeline（学習履歴層）

各Layerは異なるストレージとスキーマを持ちつつ、SubSkillタグと信頼スコア・鮮度・評価情報を介して連携する。[file:1]

### 1.2 4層モデルの目的
- 知識そのもの（What）と、実験体験（How）、評価（How well）、時系列（When）を分離する。
- 自律学習ループが「どこが弱いか」「何を学ぶべきか」「何が古いか」「何を試したか」を機械的に判断できるようにする。[file:1]

### 1.3 SubSkillとの関係
すべての層は SubSkill タグをキーとして連動する。
SubSkillは、Brain内部構造の横断的な分類軸（列）であり、4層は縦方向の観点（層）として機能する。[file:1]

## 2. 共通設計方針

### 2.1 SubSkillタグの共通利用
- 全LayerでSubSkillタグを共通の文字列IDとして利用する。
- SubSkill構造は `subskill_manifest.json` にて定義される。[file:1]

### 2.2 信頼スコア・鮮度・評価値
- 信頼スコア: Knowledge BaseおよびExperiment Logに紐づく「どれだけ信用できるか」の指標。[file:1]
- 知識の鮮度スコア: Learning Timelineに基づき、時間経過で減衰する。[file:1]
- 知識密度スコア: Self-Evaluationで管理する「どれだけ網羅されているか」の指標。[file:1]

### 2.3 データ削除の原則
- Knowledge Base: 古い情報は「deprecated」「古い情報フラグ」でマーキングし、極力削除しない。[file:1]
- Experiment Log: 成功・失敗問わず削除禁止。[file:1]
- Learning Timeline: 重複防止のためハッシュ値で管理するが、過去履歴は原則保持。[file:1]

## 3. Layer 1: Knowledge Base（知識層）

### 3.1 役割
Brainが保持する「知識そのもの」を格納する層。
自然言語テキストや構造化テキストを、ベクトル検索とグラフ構造の両面から扱えるようにする。[file:1]

### 3.2 実装
- Vector DB: FAISS（ローカル・無料）[file:1]
- Graph DB: Neo4j Community Edition（ローカル・無料）[file:1]

### 3.3 格納内容
代表的な項目:

- 学習テキスト断片（チャンク）
- SubSkillタグ
- 信頼スコア
- ソースURL / ファイルパス
- 取得日時
- 関連知識ノード（グラフ上のリンク）[file:1]

### 3.4 検索方式
- 意味的類似検索: FAISSによる埋め込みベースの検索。[file:1]
- 関係性検索: Neo4jによるノード・エッジベースの探索。[file:1]
- 両者を組み合わせたハイブリッド検索を前提とする。

### 3.5 SubSkill分類との関係
- 各知識チャンクには主SubSkill + 副SubSkill参照リンクが付与される。[file:1]
- 新規知識登録時、FAISSで類似度 > 0.92 の既存知識がある場合は重複判定し、更新 or 補完処理を行う。[file:1]

### 3.6 他Layerとの連携
- Learning Timeline: いつ・どこから知識が入ってきたかを参照する。[file:1]
- Self-Evaluation: SubSkillごとの知識密度スコア算出のベースデータとなる。[file:1]
- Experiment Log: 実験結果から導かれた知識（成功パターン・逆説的知識）を格納する。[file:1]

## 4. Layer 2: Experiment Log（実験体験層）

### 4.1 役割
「知識を使って実際に試した結果」を記録する層。
成功・失敗・部分成功を含む全ての実験を、削除禁止のログとして保持する。[file:1]

### 4.2 実装
- ストレージ: SQLite（ローカル・無料）[file:1]

### 4.3 格納内容
代表的なカラム:

- 実験ID
- 実施日時
- 対象SubSkillタグ
- 仮説テキスト
- 実験内容（何をどのように試したか）
- 結果（SUCCESS / FAILURE / PARTIAL）
- 自己分析テキスト（なぜ成功/失敗したか）
- 信頼スコア変化量（実験前後）[file:1]

### 4.4 成功・失敗の扱い
- 成功:
  - 正解パターン・根拠テキストをKnowledge Baseに格納。[file:1]
  - SubSkill信頼スコアを +0.05 加算。[file:1]
- 失敗:
  - Failure Logとして格納（削除禁止）。[file:1]
  - 「無効アプローチ」フラグを付与。[file:1]
  - 逆説的知識をKnowledge Baseへ格納。[file:1]
- 部分成功:
  - 成功部分・失敗部分を分離し、それぞれ個別に記録。
  - SubSkill信頼スコア +0.02 加算（フルの半分）。[file:1]

### 4.5 自己実験エンジンとの関係
- Self-Experimentation Engine がStepごとにExperiment Logへ書き込む。[file:1]
- 後続の学習や評価で、どの実験がどの知識に影響したかを追跡可能にする。

### 4.6 他Layerとの連携
- Knowledge Base: 実験から得た新知識をノードとして追加 or 更新。[file:1]
- Self-Evaluation: 成功・失敗結果に応じてSubSkillスコアの変化を反映。[file:1]
- Learning Timeline: 実験も「ソース種別=実験」として履歴管理する。[file:1]

## 5. Layer 3: Self-Evaluation（自己評価層）

### 5.1 役割
Brainが自分自身の知識状態・弱点・育成レベルを管理する層。
「何をどの程度知っているか」「どこが弱いか」「次に何を学ぶべきか」を数値とフラグで表現する。[file:1]

### 5.2 実装
- フォーマット: JSON + SQLite のハイブリッド。[file:1]

### 5.3 格納内容
代表的な項目:

- SubSkillごとの知識密度スコア（0.0〜1.0）
- 弱点フラグ（密度0.6以下で自動発火）[file:1]
- 補完学習優先度キュー（弱点SubSkillの学習リクエスト一覧）[file:1]
- 現在の育成Lv（Lv.1 / Lv.2 / Lv.3）[file:1]
- 最終自己評価日時
- 次回自己評価スケジュール[file:1]

### 5.4 スコアの意味
- 知識密度スコア:
  - 0.0: ほぼ何も知らない
  - 0.6未満: 弱点として補完対象（弱点フラグON）[file:1]
  - 0.8以上: Practitionerレベル相当
  - 0.95以上: Expertレベル相当
  - 0.99以上: Domain Sovereignレベル相当（維持が目標）[file:1]

### 5.5 自律学習ループとの関係
- Step1: 知識ギャップ検出で、Self-Evaluationのスコアを参照し、最も低いSubSkillを優先学習対象とする。[file:1]
- Step6: ランダム問題生成・自己回答・自己採点によりスコアを更新し、Lvを自動判定する。[file:1]

### 5.6 他Layerとの連携
- Knowledge Base: SubSkillごとの知識量・多様性・矛盾情報に基づき評価値を計算。
- Experiment Log: 実験結果による信頼スコア変化を反映。
- Learning Timeline: 新規学習・古い情報フラグを評価時に考慮可能。

## 6. Layer 4: Learning Timeline（学習履歴層）

### 6.1 役割
「いつ・どのソースから・どのSubSkillに・どの知識が入ってきたか」を時系列で管理する層。[file:1]
これにより、鮮度の低い知識や重複取得を検知し、更新・再学習の優先度付けが可能になる。[file:1]

### 6.2 実装
- ストレージ: SQLite（ローカル・無料）[file:1]

### 6.3 格納内容
代表的なカラム:

- 学習日時
- ソース種別（Web / PDF / 動画 / API / 実験 など）[file:1]
- ソースURL or ファイルパス
- 取得テキストのハッシュ値（完全重複防止）[file:1]
- 格納先SubSkillタグ
- 知識の鮮度スコア（経過時間による自動減衰）[file:1]
- 古い情報フラグ（鮮度スコア0.4以下で自動付与）[file:1]

### 6.4 鮮度スコアと古い情報フラグ
- 鮮度スコアは取得日時からの経過時間に応じて減少する。
- 0.4以下になった場合、「古い情報フラグ」を自動付与し、再学習の候補とする。[file:1]

### 6.5 自律学習ループとの関係
- 学習リソース自動探索の際、鮮度や古さを考慮した再取得戦略が取れる。
- 矛盾検出時に「古い情報」側を候補として `deprecated` マークを付けやすくなる。[file:1]

## 7. 層間関係と依存方向

### 7.1 基本的な依存関係
- Learning Timeline → Knowledge Base へ「どのソースから来たか」のメタ情報を提供。
- Experiment Log → Knowledge Base へ「実験で検証された知識」を追加・更新。
- Knowledge Base + Experiment Log + Learning Timeline → Self-Evaluation のスコア計算に影響。
- Self-Evaluation → 自律学習ループ / 自己実験エンジンの優先度設定に影響。[file:1]

### 7.2 依存方向のルール
- Self-Evaluationは他層の状態を入力として読むが、直接書き換えは行わない（評価結果のみを保持）。
- Learning Timelineは、Knowledge BaseのIDやSubSkillタグを参照するが、Knowledge Baseの中身は直接変更しない。

## 8. 自律学習・自己実験との接続

### 8.1 自律学習サイクルとのマッピング
- Step1（知識ギャップ検出）: Self-Evaluation（Layer3）を参照。[file:1]
- Step2（学習リソース自動探索）: Learning Timeline（Layer4）の鮮度とソース種別を参照。[file:1]
- Step3（データ取得・クリーニング・構造化）: Knowledge Base（Layer1）へ格納。[file:1]
- Step4（自己実験）: Experiment Log（Layer2）へ記録。[file:1]
- Step5（知識統合・再構造化）: Knowledge Base（Layer1）の矛盾検出・統合処理。[file:1]
- Step6（育成Lv自己評価）: Self-Evaluation（Layer3）更新。[file:1]

### 8.2 Self-Experimentation Engineとのマッピング
- 仮説生成: Self-Evaluation の弱点SubSkillに基づく。[file:1]
- 実験結果格納: Experiment Log（Layer2）。[file:1]
- 知識更新: Knowledge Base（Layer1）。[file:1]
- スコア更新: Self-Evaluation（Layer3）。[file:1]
- 履歴更新: Learning Timeline（Layer4、ソース種別=実験）。[file:1]

## 9. ファイルマッピングとI/O観点

### 9.1 Layerとファイルの対応
- Layer1 → `knowledge_graph.db`（Neo4j） + `vector_index.faiss`（FAISS）[file:1]
- Layer2 → `experiment_log.db`（SQLite）[file:1]
- Layer3 → `self_eval.json` + 必要に応じて補助的SQLiteテーブル[file:1]
- Layer4 → `learning_timeline.db`（SQLite）[file:1]

### 9.2 Hot-Swapとの関係
Brain Hot-Swap時には、各Layerに対応するファイルを一括コピー・ダンプ・リストアすることで、Brain内部状態を丸ごと保存・再装着する。[file:1]

## 10. 拡張・変更ポリシー

### 10.1 不変とする設計
- Brainは4層構造を維持する（Layer1〜4の役割分離）。[file:1]
- Experiment Logは削除禁止・追記専用ログとする。[file:1]
- Self-Evaluationはスコアとフラグによる自己認識を持つ。

### 10.2 拡張可能な点
- 各Layer内のスキーマ詳細（カラムの追加など）
- 鮮度スコアの具体的減衰関数
- 信頼スコア更新ロジック
- Self-Evaluationの評価アルゴリズム
