## 1. 自己実験エンジンの概要

### 1.1 コンセプト
SBAは「知識を得る」だけでなく、「実際に試して体験から学ぶ」ことを重視する。[file:1]  
自己実験エンジンは、Brain自身が仮説を立て、自分でテストし、成功・失敗の全体験をExperiment Logに永続蓄積するための仕組みである。[file:1]

### 1.2 役割
- 弱点SubSkillに対して自律的に仮説を生成する
- 仮説を検証する具体的な実験手順を設計する
- 実験を実行し、結果を評価する
- 成功・失敗・部分成功の全てをExperiment Logに記録する
- 結果をKnowledge Base・Self-Evaluationへ反映する[file:1]

## 2. 実験サイクル全体像

### 2.1 実験サイクル（高レベル）
1. 仮説生成（Step1）
2. 実験設計（Step2）
3. 実験実行（Step3）
4. 結果評価（Step4）
5. 知識更新（Step5）[file:1]

### 2.2 自律学習ループとの関係
- 自律学習ループのStep4として、自己実験エンジンが呼び出される。[file:1]
- Experiment LogはBrain内部構造のLayer2として、削除禁止の体験データを蓄積する。[file:1]

## 3. 実験種別の定義

### 3.1 種別A: 知識確認実験
- 内容: 問題生成 → 自己回答 → 正誤判定。[file:1]
- 目的: 「知っているつもり」を検証し、基本知識の穴を発見する。

### 3.2 種別B: 推論実験
- 内容: 論理問題生成 → 多段推論 → 矛盾チェック。[file:1]
- 目的: 推論能力・一貫性・矛盾検出能力を検証する。

### 3.3 種別C: コード実験（Tech系のみ）
- 内容: コード生成 → サンドボックス実行（Python subprocess）→ 結果検証。[file:1]
- 使用エンジン: Qwen2.5-Coder（Tier3） + Python subprocess。[file:1]
- 目的: 実際に動くコードを書けるか・期待通りに動くかを検証する。

### 3.4 種別D: シミュレーション実験
- 内容: 仮想ビジネスケースなどのシナリオ生成 → 判断 → 自己評価。[file:1]
- 目的: 意思決定・戦略設計系の能力を仮想環境で検証する。

## 4. Step1: 仮説生成

### 4.1 入力
- Self-Evaluation（Layer3）の弱点SubSkill情報（密度スコア・弱点フラグ）。[file:1]

### 4.2 出力
- 「〇〇のSubSkillにおいて△△を試したらどうなるか」という形式の仮説テキスト。[file:1]

### 4.3 処理概要
1. 弱点SubSkill（密度0.6以下）から対象を選定する。[file:1]
2. Tier1（Phi-4:14B）に対し、弱点SubSkillを踏まえた仮説生成プロンプトを実行する。[file:1]
3. 生成された仮説をExperiment Logの新規レコードとして準備する。

---

## 5. Step2: 実験設計

### 5.1 入力
- 仮説テキスト
- 対象SubSkill
- 実験種別（A/B/C/D）選択条件

### 5.2 出力
- 実験種別（A/B/C/D）
- 具体的な実験手順（プロンプト・コード・シナリオ）

### 5.3 実験種別選択ロジック
- 知識確認が主目的 → 種別A
- 論理・推論能力の検証 → 種別B
- Tech系Brain + コード実験フェーズ → 種別C
- ビジネス/戦略/意思決定系 → 種別D[file:1]

### 5.4 実験手順生成
- Tier1（Phi-4:14B）に対し、「この仮説を検証するための実験手順を設計せよ」とプロンプトを投げ、手順を生成する。[file:1]
- 種別Cでは、Qwen2.5-Coder向けのコード生成プロンプトも含めて出力させる。[file:1]

---

## 6. Step3: 実験実行

### 6.1 種別A/B（知識確認・推論実験）
- 実行エンジン: Tier1（Phi-4:14B）。[file:1]
- 処理:
  - 自身で問題を生成し、自身で回答し、自身で採点する。
  - 採点結果（正答/誤答/部分正答）を記録する。

### 6.2 種別C（コード実験）
- 実行エンジン: Qwen2.5-Coder（Tier3） + Python subprocess。[file:1]
- 処理:
  1. Qwen2.5-Coderがコードを生成する。[file:1]
  2. 生成コードをサンドボックス環境（subprocess）で実行。[file:1]
  3. 標準出力・標準エラー・終了コードを取得し、期待結果と比較する。

### 6.3 種別D（シミュレーション実験）
- 実行エンジン: Tier1（Phi-4:14B）。[file:1]
- 処理:
  - 仮想ビジネスケースやシナリオを生成し、その中で自分の判断をシミュレートする。
  - 自己評価基準に基づき、シナリオ内での判断を採点する。

### 6.4 Experiment Logへの記録
- 実験ID、SubSkill、仮説、手順、入力、出力、ログ、実行時間などをExperiment Log（SQLite）に記録する。[file:1]

---

## 7. Step4: 結果評価

### 7.1 成功時の処理
- 条件: 実験結果が期待通り、または許容範囲内であると判断された場合。
- 処理:[file:1]
  1. 正解パターン・根拠テキストをKnowledge Base（Layer1）に格納。[file:1]
  2. 対象SubSkillの信頼スコアを +0.05 加算。[file:1]
  3. Experiment Logに結果=SUCCESSとして記録。[file:1]

### 7.2 失敗時の処理
- 条件: 実験が期待通りに動作しなかった、または誤答・矛盾が発生した場合。
- 処理:[file:1]
  1. エラーパターン・失敗原因をFailure LogとしてExperiment Logに格納（削除禁止）。[file:1]
  2. 「このアプローチは無効」フラグを付与して保管。[file:1]
  3. なぜ失敗したかの自己分析テキストを生成して記録。[file:1]
  4. 失敗から導き出せる「逆説的な知識」（やってはいけない方法など）をKnowledge Baseへ格納。[file:1]
  5. 対象SubSkillの信頼スコアは変更しない（失敗情報は別管理）。[file:1]

### 7.3 部分成功時の処理
- 条件: 一部は期待通りだが、一部に問題がある場合。
- 処理:[file:1]
  1. 成功部分・失敗部分を分離し、それぞれに上記成功/失敗処理を適用。
  2. SubSkill信頼スコアを +0.02 加算（フルスコアの半分）。[file:1]

## 8. Step5: 知識更新・グラフ反映

### 8.1 Knowledge Baseへの反映
- 実験により検証された知識ノードの信頼スコアを更新する。[file:1]
- 仮説が証明された場合は、当該知識ノードの信頼スコアをUP。[file:1]
- 仮説が否定された場合は、「実験否定済み」フラグを付与する。[file:1]

### 8.2 Neo4jグラフ更新
- 新しい知識ノードや関係性（EDGE）を追加・更新する。[file:1]
- 逆説的知識を含むノードにも適切なラベルやフラグを付与する。

### 8.3 Self-Evaluationへの影響
- 実験結果に基づいて、SubSkillの知識密度スコア・信頼スコアの調整材料として利用する。

## 9. 実験スケジューラ設計

### 9.1 軽量実験（種別A/B）
- 実行間隔: 1時間ごと。[file:1]
- 実行時間: 約5〜15分。[file:1]
- 実験数: 1回あたり3〜5問。[file:1]
- 目的: 日常的な知識確認・推論テスト。

### 9.2 中量実験（種別C/D）
- 実行間隔: 6時間ごと。[file:1]
- 実行時間: 約30〜60分。[file:1]
- 実験数: 1回あたり1〜2ケース。[file:1]
- 目的: コード検証・複雑なシミュレーション。

### 9.3 重量実験（総合自己評価）
- 実行間隔: 24時間ごと（深夜自動実行推奨）。[file:1]
- 実行時間: 約2〜4時間。[file:1]
- 内容: 全SubSkillへのランダム問題出題・回答・採点・Lv再評価、知識グラフ再構造化。[file:1]

### 9.4 実装（APScheduler想定）
- 軽量 / 中量 / 重量それぞれをAPSchedulerのジョブとして登録する。
- SQLite JobStoreでジョブ定義を永続化し、再起動後もスケジュールを維持する。

## 10. リソース・VRAM・安全性

### 10.1 VRAM制約とTier3利用
- 種別C（コード実験）でQwen2.5-Coder（Tier3）を起動する際、Tier1との同時起動は原則禁止（VRAM競合）。[file:1]
- 実験実行前に、必要なOllamaモデルのロード/アンロードを制御する。

### 10.2 サンドボックス実行の安全性
- subprocess実行はサンドボックス（専用一時ディレクトリ・制限付きPython）で行う。
- タイムアウト・リソース上限・外部アクセス禁止などのガードを設ける。

### 11.1 Experiment Logの位置付け
- Brain内部構造のLayer2として、全ての実験体験を永続保存し、削除禁止とする。[file:1]

### 11.2 記録内容（再掲）

- 結果（SUCCESS / FAILURE / PARTIAL）
- 自己分析テキスト
- 関連SubSkillタグ
- 信頼スコア変化量[file:1]

## 12. 関連コンポーネントとの関係

### 12.1 自律学習ループ
- 自律学習ループのStep4を担う。
- Step1の弱点検出・Step6の自己評価と密接に連携する。[file:1]

### 12.2 Brain内部構造
- Layer2（Experiment Log）を直接操作し、Layer1（Knowledge Base）とLayer3（Self-Evaluation）に影響を与える。[file:1]

### 12.3 推論エンジン
- Tier1: 仮説生成・実験設計・知識確認・推論・シミュレーションに使用。[file:1]
- Tier3: コード実験時のコード生成に使用。[file:1]

## 13. 不変事項と拡張余地

### 13.1 不変事項
- 成功・失敗を問わず、全実験ログを削除禁止で永続保存する。[file:1]
- 種別A/B/C/Dの4種構造を維持する（追加は可だが削除・統廃合は慎重に）。[file:1]
- 成功時+0.05、部分成功+0.02のスコア加算は基準値とする。[file:1]

### 13.2 拡張可能事項
- 実験種別の追加（例: 現場データ再現実験など）
- スコア加算値のチューニング
- 実験設計アルゴリズムの高度化（メタ実験など）

## 14. 付録

### 14.1 高レベル擬似コード

```python
def run_self_experiments(brain):
    weak_subskill = select_weak_subskill(brain.self_eval)
    hypothesis = generate_hypothesis(weak_subskill)
    exp_type, plan = design_experiment(hypothesis, weak_subskill)
    result = execute_experiment(exp_type, plan)
    eval_info = evaluate_result(result)
    update_knowledge_and_logs(brain, weak_subskill, hypothesis, result, eval_info)
```

### 14.2 テスト観点メモ
- 連続して実験を行った際、Experiment Logが正しく増え続けるか
- 失敗実験が知識に悪影響を与えず、逆説的知識として整理されるか
- コード実験がサンドボックス外へ影響を及ぼさないか
