## 1. 境界と前提

### 1.1 SBAの責務範囲
- SBAの責務: Brainの育成・保管・Hot-Swap・エクスポートまで。[file:1]
- SBAが行わないこと:
  - 実務処理（文書作成・コード実行・業務判断・返信送付等）。[file:1]
  - 実務ツールとの直接通信・連携・API呼び出し。[file:1]
  - ユーザーへの実務アウトプット提供。[file:1]

### 1.2 外部ツールの責務範囲
- エクスポート済みBrainを読み込み、実務ワークフローの中で活用する。
- 実務ツールはSBAプロセスに依存せず、brain_api.pyとエクスポートディレクトリのみで完結して動作する。[file:1]

## 2. エクスポート出力物

### 2.1 ディレクトリ構成
エクスポートコマンドの出力構成は以下とする。[file:1]

```text
C:/exports/Python開発_v1.4/
├── knowledge_graph.db         ← Neo4j dump（読み取り専用）
├── vector_index.faiss         ← FAISSインデックス（読み取り専用）
├── experiment_log.db          ← 成功/失敗体験DB（読み取り専用）
├── subskill_manifest.json     ← SubSkill構造・タグマップ
├── metadata.json              ← ドメイン・バージョン・達成Lv・エクスポート日時
└── brain_api.py               ← 実務ツール用APIラッパー（SBA依存なし・スタンドアロン動作）
```
[file:1]

### 2.2 各ファイルの役割（外部利用視点）
- knowledge_graph.db:
  - グラフ検索・関係性ベースの参照用。[file:1]
- vector_index.faiss:
  - 意味類似検索（質問→関連チャンク抽出）用。[file:1]
- experiment_log.db:
  - 成功/失敗体験の参照用（「過去にどう試したか」）。[file:1]
- subskill_manifest.json:
  - SubSkill一覧・タグの意味解釈用。[file:1]
- metadata.json:
  - ドメイン名・バージョン・レベル・エクスポート日時の参照用。[file:1]
- brain_api.py:
  - 外部からBrainを操作する唯一のインターフェース。[file:1]

## 3. エクスポートコマンド仕様（SBA側）

### 3.1 コマンド形式
```bash
sba brain export "Python開発_v1.4" --output C:/exports/
```
[file:1]

### 3.2 処理フロー概要
1. Brain Bank から指定Brainディレクトリ（例: `Python開発_v1.4/`）を読み込む。[file:1]
2. 上記5ファイル（knowledge_graph.db, vector_index.faiss, experiment_log.db, subskill_manifest.json, metadata.json）を出力先へコピーする。[file:1]
3. 出力先に brain_api.py を生成する（SBAへの依存を持たないスタンドアロン実装）。[file:1]
4. metadata.json に `exported_at` を追記/更新する。

---

## 4. 読み取り専用ポリシー

### 4.1 外部利用時の制約
- knowledge_graph.db / vector_index.faiss / experiment_log.db は読み取り専用として扱う。[file:1]
- 実務ツールが知識を更新・追加したい場合は、SBA側プロセスとは切り離した別ストアで行う（本仕様書の対象外）。

### 4.2 書き込みの扱い
- brain_api.py 経由での操作は原則「読み取りAPI」のみを提供する。
- 書き込みAPIは、SBAへの逆輸入やBrain構造破壊を防ぐため、本仕様では提供しない。

## 5. brain_api.py の責務

### 5.1 役割
- エクスポートされたBrainを外部アプリから簡便に扱うためのPythonクライアント。
- SBA本体やOllama等には依存せず、エクスポートディレクトリ内のファイルのみで完結する。[file:1]

### 5.2 利用イメージ
```python
from brain_api import BrainClient

brain = BrainClient("C:/exports/Python開発_v1.4")

result = brain.query(
    question="FastAPIのルーター設計のベストプラクティスを教えて",
    subskill="設計",
    top_k=5
)

info = brain.info()
scores = brain.eval_scores()
all_knowledge = brain.list_subskill("デバッグ")
```
[file:1]

## 6. brain_api.py インターフェース定義（外形）

### 6.1 BrainClient クラス

#### 6.1.1 コンストラクタ
```python
BrainClient(path: str)
```
- `path`: エクスポートされたBrainディレクトリへのパス（末尾スラッシュなし）。[file:1]

#### 6.1.2 メソッド一覧
- `query(question: str, subskill: str | None = None, top_k: int = 5) -> list[dict]`
- `list_subskill(subskill_id: str) -> list[dict]`
- `info() -> dict`
- `eval_scores() -> dict`

### 6.2 query メソッド（概要仕様）
- 入力:
  - `question`: ユーザー質問文。
  - `subskill`: 絞り込み対象SubSkill（省略可）。[file:1]
  - `top_k`: 返却する候補数。
- 処理:
  - vector_index.faiss から意味類似検索で関連チャンクを抽出。
  - knowledge_graph.db で関連ノード・コンテキストを補完。
  - experiment_log.db から関連する成功/失敗体験を添付して返却。
- 出力:
  - 各候補ごとにテキスト、SubSkill、信頼スコア、ソース情報などを含む辞書リスト。

### 6.3 list_subskill メソッド（概要仕様）
- 入力: `subskill_id`（例: "デバッグ"）。[file:1]
- 処理: 当該SubSkillに属する知識チャンク全件を列挙。
- 出力: テキスト・信頼スコア・ソースURL等のリスト。

### 6.4 info メソッド（概要仕様）
- 出力例:
```json
{
  "domain": "Python開発",
  "version": "1.4",
  "level": 2,
  "subskills": ["設計", "実装", "デバッグ", "修正", "テスト", "レビュー", "ドキュメント"]
}
```
[file:1]

### 6.5 eval_scores メソッド（概要仕様）
- 出力:
  - SubSkillごとの信頼スコア・知識密度スコアなどを返す（self_eval.jsonベース）。[file:1]

## 7. データアクセス仕様

### 7.1 Neo4j dump の扱い
- エクスポートされた knowledge_graph.db は、スタンドアロンのGraphクエリ層から読み取り専用で参照する。
- 利用側でNeo4jサーバを立てる / ローカルで一時起動するかはbrain_api.py 実装仕様で定義。

### 7.2 FAISS インデックスの扱い
- vector_index.faiss はメモリにロードして検索可能にする。
- top_k 検索は query メソッド内で wrap する。

### 7.3 SQLite DB の扱い
- experiment_log.db, learning_timeline.db は read-only モードで開く。
- experiment_log.db はbrain_api.pyが必要に応じて関連実験を引くために利用する。[file:1]

## 8. 外部ツール統合パターン

### 8.1 典型統合フロー
1. SBAからBrainをエクスポート（C:/exports/...）。[file:1]
2. 実務ツールプロジェクト内に `brain/` ディレクトリを作成し、エクスポートディレクトリをコピー or マウントする。
3. 実務ツールコードから `brain_api.py` をimportし、BrainClientを初期化する。
4. UIや業務ロジックからBrainClient.query/info等を呼び出して知識参照を行う。

### 8.2 マルチBrain利用
- 複数Brain（例: Python開発, 法人営業）を同一ツールから扱う場合、BrainClientインスタンスを複数作成するか、簡易ラッパでBrainを切り替える。

## 9. セキュリティ・アクセス制御

### 9.1 権限の前提
- エクスポートされたBrainにはアクセス制御機能を内包しない（純粋なデータパッケージ）。
- アクセス制御は実務ツール側の責務とする。

### 9.2 データ保護
- 秘匿性の高いBrain（たとえば企業固有の営業ノウハウ等）は、エクスポート後の保存場所と権限管理を実務ツール側で徹底する。

## 10. バージョン互換性

### 10.1 metadata.json による管理
- metadata.jsonの `version` によりBrainのスキーマバージョンを識別する。[file:1]
- brain_api.py は、自身が対応する最低/最大バージョンレンジを内部に持ち、非対応の場合は明示的に例外を投げる。

### 10.2 後方互換ポリシー
- 可能な限り後方互換を維持し、軽微なスキーマ変更はbrain_api.py側で吸収する。
- 破壊的変更が必要な場合、新しい brain_api_X.Y.py として別名で出力することも検討する。

---

## 11. SBA側「エクスポートまで」の責務

### 11.1 保証すること
- Brain Bank上のBrainと、エクスポートされたBrainが論理的に同一状態であること（ファイルコピー整合）。[file:1]
- metadata.json に正しいドメイン・バージョン・レベル・エクスポート日時が記録されていること。[file:1]

### 11.2 保証しないこと
- 実務ツールでの利用結果（そのBrainを使ってどういう業務判断が行われたか）。
- 実務ツールにおけるバグや誤用。

## 12. 実務ツール側ガイド（概要）

### 12.1 やってよいこと
- BrainClient.query で知識チャンクを取得し、自分のLLMやテンプレートエンジンと組み合わせて回答を生成する。
- 特定SubSkillの知識リストをUIに表示し、ナレッジブラウザとして使う。

### 12.2 やらないほうがよいこと
- エクスポートディレクトリ内のDBやインデックスを書き換える。
- metadata.json / subskill_manifest.jsonを直接編集してBrainの意味構造を変える。

## 14. 付録

### 14.1 最小統合理コード例（擬似）

```python
from brain_api import BrainClient

brain = BrainClient("C:/exports/Python開発_v1.4")

q = input("質問を入力してください: ")
hits = brain.query(q, top_k=5)

for i, h in enumerate(hits, 1):
    print(f"[{i}] score={h['score']:.2f} subskill={h['subskill']}")
    print(h['text'])
    print("---")
```
