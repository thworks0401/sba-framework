# 育成ドメインカタログ・テンプレート設計書（Brain量産のためのドメイン雛形）

---

## 0. 文書情報

### 0.1 文書名
育成ドメインカタログ・テンプレート設計書（Brain量産のためのドメイン雛形）

### 0.2 目的
本書は、SBAで育成可能なBrainドメインのカタログと、その定義テンプレート（名称・カテゴリ・目的・対象ユーザー・初期SubSkill雛形など）を定め、Brainを量産する際の基準と雛形を提供することを目的とする。[file:1]

### 0.3 本書の位置付け
- 上位資料: SBAコンセプト・境界設定書、Brain内部構造設定書
- 関連資料: SubSkill体系設定書、Brain Package・保存形式設定書、成長評価・Lv定義設定書
- 下位資料: 個別Brain設計書（例: Python開発Brain設計書、法人営業Brain設計書）

### 0.4 想定読者
- アーキテクト / 設計担当
- Brain設計・運用担当
- 将来のBrain追加時の設計者

---

## 1. ドメインカタログの考え方

### 1.1 ドメインとは
本書における「ドメイン」とは、1つのBrainが専門として育成される知識・スキル領域（例: Python開発、法人営業、青色申告、不動産投資など）を指す。[file:1]

### 1.2 カタログの目的
- どの分野のBrainを育成対象とするかを一覧化する。
- 各ドメインの**目的・対象ユーザー・初期SubSkill構造**を雛形として定義し、Brain量産時のぶれを防ぐ。[file:1]

### 1.3 カテゴリ構造
ドメインはカテゴリごとにグルーピングする。[file:1]  
例: Tech/開発系、AI/データ系、事務/総務系、営業/セールス系、企業経営系、マーケ系、財務/税務系、法律系、人事系、PM系、クリエイティブ系、学術系、投資系、EC系、SaaS/ITビジネス系 など。[file:1]

---

## 2. ドメイン定義テンプレート

### 2.1 テンプレート項目一覧
各Brainドメインは最低限以下の項目で定義する。

- ドメインID（英数字）
- ドメイン名（日本語）
- カテゴリ（上記カテゴリのいずれか）
- 概要（1〜2行）
- 目的（Brainとして何ができるようになるか）
- 対象ユーザー（想定利用者像）
- 初期SubSkill雛形（名前＋簡単な説明）
- 優先育成度（High/Medium/Low）
- 備考

### 2.2 テンプレート例（Python開発 Brain）[file:1]

```yaml
domain_id: python_dev
domain_name: Python開発
category: Tech/開発系
summary: Pythonを用いたアプリケーション・スクリプト・API開発全般に関するBrain。
purpose: >
  要件定義〜実装〜テスト〜運用まで、Python開発に必要な知識と実務パターンを網羅し、
  現役エンジニア同等以上の設計・実装支援ができる状態を目指す。
target_users:
  - 個人開発者
  - 受託開発エンジニア
  - 社内ツール開発担当
initial_subskills:
  - id: 設計
    description: アーキテクチャ・要件定義・設計パターン・クラス設計
  - id: 実装
    description: コーディング規約・実装手法・ライブラリ活用
  - id: デバッグ
    description: エラーパターン・スタックトレース解析・ログ調査
  - id: 修正
    description: リファクタリング・コード改善・パフォーマンス最適化
  - id: テスト
    description: ユニットテスト・結合テスト・カバレッジ管理
  - id: レビュー
    description: コードレビュー基準・品質指標・セキュリティチェック
  - id: ドキュメント
    description: README・docstring・API仕様書作成
priority: High
notes: 初期の育成推奨Brain。[1]
```

### 2.3 テンプレート例（法人営業 Brain）[file:1]

```yaml
domain_id: corporate_sales
domain_name: 法人営業
category: 営業/セールス系
summary: BtoB法人営業の戦略設計〜案件獲得〜フォローまでを扱うBrain。

  ターゲット設定からヒアリング・提案・クロージング・フォローまで、
  再現性の高い法人営業プロセスをBrainとして体系化する。

  - 法人営業担当
  - インサイドセールス
  - セールスマネージャー

  - id: 顧客開拓
    description: リスト作成・ターゲット設定・アプローチ設計
  - id: ヒアリング
    description: 課題抽出・ニーズ把握・質問設計
  - id: 提案設計
    description: 提案書構成・ソリューション設計・価格提示

    description: 交渉・合意形成・反論処理テクニック

    description: 契約書確認・条件整理・捺印フロー
  - id: フォロー
    description: 関係維持・アップセル・クレーム対応
priority: Medium
notes: サブスク営業Brain等と連携することで応用可。[1]
```

## 4. Brain量産フローとこのカタログの使い方

### 4.1 新規Brain追加フロー
1. カタログから候補ドメインを選ぶ or 新規ドメインを定義する。[file:1]  
2. 本書のテンプレに沿ってYAML雛形を作成し、`domains/` ディレクトリに保存する。  
3. SubSkill体系設定書を参照し、initial_subskillsを精査する。  
4. Blank Templateを複製し、metadata.jsonにdomain名を書き込んだBrainを生成する。[file:1]

### 4.2 優先度管理
- `priority` フィールド（High/Medium/Low）で育成順を管理する。
- たとえば「Python開発」「青色申告」「AIエージェント設計」はHighなど。[file:1]

---

## 5. 不変事項と拡張余地

### 5.1 不変事項
- ドメイン定義は「ドメイン名＋初期SubSkill雛形＋目的」を必ず含む。[file:1]
- SubSkill構造の詳細は別紙（SubSkill体系設定書）で管理し、本書はあくまで雛形とする。[file:1]

### 5.2 拡張余地
- ドメインごとに「推奨データソース」「推奨自己実験種別」などを付け足して、より細かいテンプレにしていく。
- 「ビジネス優先度」「市場性」「実務連携のしやすさ」などの評価軸を追加する。
